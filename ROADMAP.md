## Что нужно сделать

### Принцип: сначала наш бэкенд и моки, потом внешний API

- **Наши данные** (баланс, уровни, квесты, магазин, транзакции, штрафы, заявки) — сразу хранятся в нашей БД и отдаются нашим API.
- **Данные основной системы** (смены, отмены, прогулы, профиль пользователя, авторизация) — до подключения реального API используем **плейсхолдеры/моки**:
  - тестовые пользователи и «заглушка» авторизации;
  - ручной ввод или админские действия вместо событий «смена завершена» / «отмена» / «no-show»;
  - при необходимости — мок-сервис или конфиг с тестовыми сменами.
- **Этап интеграции с API основной системы** — в конце: замена заглушек на реальные вызовы и приём событий.

### Этап 1: Наш бэкенд и база данных

Без обращения к внешнему API. Все данные — в нашей БД. **Бэкенд разрабатывается на базе шаблона микросервиса `nestjs-service/`** (см. ARCHITECTURE.md, п. 1.1).

| № | Задача | Детали | Статус |
|---|--------|--------|--------|
| 1.0 | **Использовать шаблон nestjs-service** | Скопировать/клонировать папку `nestjs-service/`, выполнить `npm i` и `./init.sh --name "MyGig Rewards API" --description "Сервис наград и лояльности"`. Дальнейшую разработку вести в этом проекте: добавлять модули, контроллеры, схемы БД. | ⏳ Выполнить `init.sh` (на Windows — вручную заменить плейсхолдеры или через Git Bash) |
| 1.1 | Стек и структура | Опираться на шаблон: NestJS, Drizzle, PostgreSQL, Redis, Auth, Health, Swagger, Helm. При необходимости отключить или заменить неиспользуемые части (например Mongoose), сохраняя общую структуру. | ✅ |
| 1.2 | База данных | Схема по ARCHITECTURE.md в Drizzle: users, levels, strikes, transactions, store_items, redemptions, quests, quest_progress, audit_log. Использовать существующие в шаблоне миграции и `base.schema` (timestamps). | ✅ Реализовано в `nestjs-service/src/infra/db/drizzle/schemas/` |
| 1.3 | Миграции | Генерация и применение миграций через скрипты шаблона (`db:generate`, `db:migrate`), первичное заполнение справочника уровней. | ✅ Добавлена миграция `0001_rewards_schema.sql` и сидер уровней; для применения: задать `PG_CONNECTION`/`DATABASE_URL` и `MIGRATIONS_PATH=./migrations`, затем `npm run db:migrate:local` |
| 1.4 | API для дашборда | Эндпоинты: текущий пользователь (профиль, баланс, уровень, штрафы за 30 дней), история транзакций, квесты с прогрессом, каталог магазина, создание заявки на обмен. Для разработки: query-параметр `?userId=` или `DEV_USER_ID` в env. |

**Плейсхолдер:** «текущий пользователь» отдаётся по идентификатору из сессии/токена; до этапа авторизации можно использовать фиксированный test user id из конфига или query-параметр для разработки.

### Этап 4: Механики на наших данных и моках событий

Вся логика уровней, квестов и штрафов работает в нашем бэкенде. События «смена завершена», «отмена», «no-show» **не** приходят из внешнего API — имитируем их.

| № | Задача | Детали | Статус |
|---|--------|--------|--------|
| 4.1 | Начисление монет за смену | Эндпоинт или внутренний сервис: «засчитать завершённую смену» для пользователя (user_id, сумма монет, описание). Обновление баланса, транзакция, увеличение shifts_completed. **Мок:** вызов вручную (админ или dev-инструмент) или тестовый скрипт. | ✅ `POST /v1/rewards/shifts/complete` (body: `coins`, `title?`, `location?`). Обновление баланса, транзакция type=shift, +1 shifts_completed, пересчёт уровня. |
| 4.2 | Пересчёт уровня | При изменении shifts_completed пересчитывать уровень пользователя по справочнику уровней. | ✅ Выполняется в `recordShiftCompleted` и при необходимости вызывается отдельно `recalcUserLevel(userId)`. Уровень = макс. по shifts_required, где shifts_required <= user.shifts_completed. |
| 4.3 | Квесты | Логика условий (смены за день/неделю, серия, ранняя смена, рейтинг). Прогресс хранить в quest_progress; при выполнении — начисление монет и транзакция. **Мок:** данные для проверки условий (число смен, время смены, рейтинг) пока вводим вручную или через админку/сидеры, без API основной системы. | ⏳ Прогресс квестов отдаётся с API; логика начисления за выполнение квеста (обновление quest_progress.completedAt, транзакция) — при необходимости в джобе или при вызове «засчитать смену». |
| 4.4 | Штрафы и понижение уровня | Эндпоинт или внутренний сервис: «зарегистрировать штраф» (user_id, тип: late_cancel / no_show). Запись в strikes, подсчёт за 30 дней, при достижении порога уровня — понижение. **Мок:** вызов вручную (админ или dev) вместо реальных событий от основной системы. | ✅ `POST /v1/rewards/strikes` (body: `type: "no_show" \| "late_cancel"`, `shiftExternalId?`). Запись в strikes, подсчёт за 30 дней, при count >= strike_threshold текущего уровня — понижение на 1 уровень. |
| 4.5 | Отображение штрафов в UI | Показ «N/M штрафов (до понижения)» на главной или в блоке уровня (данные с нашего API). | ✅ В блоке «Прогресс уровня» выводится строка «N/M штрафов за 30 дней (до понижения)»; при N >= M — подсветка (до понижения). Для бронзы (порог null) строка не показывается. |

После этапа интеграции с API основной системы вызовы «засчитать смену» и «зарегистрировать штраф» будут вызываться автоматически при получении событий из внешней системы.
