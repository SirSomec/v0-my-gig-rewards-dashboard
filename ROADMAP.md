## Что нужно сделать

### Принцип: сначала наш бэкенд и моки, потом внешний API

- **Наши данные** (баланс, уровни, квесты, магазин, транзакции, штрафы, заявки) — сразу хранятся в нашей БД и отдаются нашим API.
- **Данные основной системы** (смены, отмены, прогулы, профиль пользователя, авторизация) — до подключения реального API используем **плейсхолдеры/моки**:
  - тестовые пользователи и «заглушка» авторизации;
  - ручной ввод или админские действия вместо событий «смена завершена» / «отмена» / «no-show»;
  - при необходимости — мок-сервис или конфиг с тестовыми сменами.
- **Этап интеграции с API основной системы** — в конце: замена заглушек на реальные вызовы и приём событий.

### Этап 4: Механики на наших данных и моках событий

Вся логика уровней, квестов и штрафов работает в нашем бэкенде. События «смена завершена», «отмена», «no-show» **не** приходят из внешнего API — имитируем их.

| № | Задача | Детали | Статус |
|---|--------|--------|--------|
| 4.1 | Начисление монет за смену | Эндпоинт или внутренний сервис: «засчитать завершённую смену» для пользователя (user_id, сумма монет, описание, опционально клиент/категория/часы). Обновление баланса, транзакция, увеличение shifts_completed. **Мок:** вызов вручную (админ или dev-инструмент) или тестовый скрипт. | ✅ `POST /v1/rewards/shifts/complete` (body: `coins`, `title?`, `location?`, `clientId?`, `category?`, `hours?`). Обновление баланса, транзакция type=shift с полями client_id, category, hours, +1 shifts_completed, пересчёт уровня и прогресса квестов. |
| 4.2 | Пересчёт уровня | При изменении shifts_completed пересчитывать уровень пользователя по справочнику уровней. **Переход на следующий уровень требует выполнения порога смен именно следующего уровня.** Базовый уровень (минимальный sort_order) выдаётся изначально без условий (порог 0); при отсутствии подходящего уровня по сменам назначается базовый. Обновление только при **повышении** (по sort_order), чтобы не затирать ручное назначение админом. При любой смене уровня (повышение, понижение по штрафам, ручное в админке) счётчик смен **обнуляется**. | ✅ Выполняется в `recordShiftCompleted` и в начале `getMe` (`recalcUserLevel`). Уровень = макс. по shifts_required, где shifts_required <= user.shifts_completed. Fallback на базовый уровень при отсутствии подходящего. API `GET /v1/rewards/me` возвращает `nextLevelShiftsRequired` — порог следующего уровня для отображения прогресса. |
| 4.3 | Квесты | Логика условий (смены за день/неделю/месяц, по клиенту, по категории, часы). Прогресс хранить в quest_progress; при выполнении — начисление монет и транзакция. **Мок:** данные для проверки условий (число смен, часы, client_id, category) вводятся при записи смены в админке. | ✅ Условия: `shifts_count`, `shifts_count_client`, `shifts_count_clients`, `shifts_count_category`, `hours_count`, `hours_count_client`, `hours_count_clients`. Транзакции с полями client_id, category, hours. `recalcQuestProgressForUser` в конце `recordShiftCompleted`; при достижении цели — начисление монет (транзакция type=quest). Периоды daily/weekly/monthly; единоразовый признак; автоотключение по active_until. |
| 4.4 | Штрафы и рейтинг надёжности | Эндпоинт или внутренний сервис: «зарегистрировать штраф» (user_id, тип: late_cancel / no_show). Запись в strikes; **снижение рейтинга надёжности** пользователя на величину из настроек (отдельные параметры для прогула и поздней отмены). Уровень по штрафам не понижается. **Мок:** вызов вручную (админ или dev) вместо реальных событий от основной системы. | ✅ `POST /v1/rewards/strikes` (body: `type: "no_show" \| "late_cancel"`, `shiftExternalId?`). Запись в strikes, снижение `users.reliability_rating` по типу штрафа (настройки в system_settings). |
| 4.5 | Отображение рейтинга в UI | Показ **рейтинга надёжности** (0–5, дробное) в блоке «Прогресс уровня»: значение X.X / 5 и шкала; подпись о том, что рейтинг растёт за смены и снижается за прогулы/поздние отмены. | ✅ В блоке «Прогресс уровня» отображается «Рейтинг надёжности» с шкалой 0–5. Данные с `GET /v1/rewards/me` (reliabilityRating). Настройки рейтинга — в админке «Настройки» (прирост за смену, снижение за прогул, снижение за позднюю отмену). |

После этапа интеграции с API основной системы вызовы «засчитать смену» и «зарегистрировать штраф» будут вызываться автоматически при получении событий из внешней системы.

| 6.9 | Аудит | Лог изменений (кто, когда, что). | ✅ Реализовано: таблица `audit_log` (admin_id, action, entity_type, entity_id, old_values, new_values, created_at). Запись в лог при: изменении уровня пользователя (user_level_change), ручном начислении/списании (manual_transaction), снятии штрафа (strike_removed). API: `GET /v1/admin/audit-log?page=&pageSize=&action=&entityType=` с пагинацией и фильтрами. Страница `/admin/audit` — таблица записей с фильтрами по действию и сущности. |
| 6.10 | Имитация событий смен (мок) | В админке или отдельном dev-инструменте: кнопки/формы «Засчитать смену», «Зарегистрировать позднюю отмену», «Зарегистрировать прогул» для выбранного пользователя — вызывают ту же логику, что потом будет вызываться из webhook/API основной системы. | ✅ Страница `/admin/dev`: форма «Записать смену» (userId, монеты, название) → `POST admin/shifts/complete`; форма «Добавить штраф» (userId, тип no_show/late_cancel) → `POST admin/strikes`. |
