## Что нужно сделать

### Этап 1: Бэкенд и данные

| № | Задача | Детали |
|---|--------|--------|
| 1.1 | Выбор стека бэкенда | Node.js (Next.js API Routes или отдельный сервис), Python (FastAPI), Go и т.д. |
| 1.2 | База данных | Выбор СУБД (PostgreSQL рекомендована в ARCHITECTURE.md), схема таблиц по ARCHITECTURE.md (users, levels, strikes, transactions, store_items, redemptions, quests, quest_progress, audit_log). |
| 1.3 | Миграции | Инструмент миграций (например Prisma, Drizzle, Alembic), первичное заполнение справочника уровней. |
| 1.4 | API для дашборда | Эндпоинты: текущий пользователь (профиль, баланс, уровень, штрафы за 30 дней), история транзакций, список квестов с прогрессом, каталог магазина, заявка на обмен. |

### Этап 2: Авторизация

| № | Задача | Детали |
|---|--------|--------|
| 2.1 | Интеграция с основной системой | Получение/проверка токена (JWT или OAuth2), маппинг external_id → наш user_id. |
| 2.2 | Сессии | Хранение сессии на фронте (cookie/httpOnly или токен в памяти), передача в заголовке запросов к нашему API. |
| 2.3 | Первый вход | Создание записи пользователя в нашей БД при первом логине (external_id, стартовый уровень). При необходимости — подтягивание имени/аватара из API основной системы. |

### Этап 3: Живые данные и механики

| № | Задача | Детали |
|---|--------|--------|
| 3.1 | Подключение фронта к API | Замена моков в `page.tsx` на вызовы нашего API (fetch/React Query/SWR), обработка загрузки и ошибок. |
| 3.2 | Начисление монет за смены | Получение событий «смена завершена» из основной системы (webhook или опрос API), расчёт суммы монет, запись транзакции, обновление баланса и shifts_completed. |
| 3.3 | Уровни | Пересчёт уровня пользователя при изменении shifts_completed (сравнение с порогами из справочника уровней). |
| 3.4 | Квесты | Логика выполнения условий (смены за день/неделю, серия, ранняя смена, рейтинг); обновление progress, при выполнении — начисление монет и запись транзакции. |
| 3.5 | Штрафы и понижение уровня | Получение событий отмены (&lt;24 ч) и no-show от основной системы; создание strike, подсчёт штрафов за 30 дней, понижение уровня при достижении порога (по справочнику уровней). |
| 3.6 | Отображение штрафов в UI | Показ «N/M штрафов (до понижения)» на экране уровня/главной (опционально). |

### Этап 4: Магазин и обмен

| № | Задача | Детали |
|---|--------|--------|
| 4.1 | Списание и заявка | При «Купить»: проверка баланса, списание монет, создание транзакции и заявки на обмен (status: pending). |
| 4.2 | Обработка заявок | Для админа: список заявок, смена статуса (fulfilled/cancelled), при отмене — возврат монет (опционально). |
| 4.3 | Выдача наград | Ручная выдача (код скидки, мерч) или интеграция с партнёрами (вне текущего плана — по необходимости). |

### Этап 5: Админ-панель

| № | Задача | Детали |
|---|--------|--------|
| 5.1 | Роут и доступ | Отдельный роут `/admin`, проверка роли admin при входе. |
| 5.2 | Управление магазином | CRUD товаров: название, описание, категория, цена, лимит, даты, видимость по группам, порядок. |
| 5.3 | Управление квестами | CRUD квестов: период (daily/weekly), условие, награда, привязка к группам; планирование на периоды. |
| 5.4 | Справочник уровней | Редактирование уровней: порог смен, порог штрафов, перки. |
| 5.5 | Пользователи | Поиск по ID/email/имени, карточка: баланс, уровень, смены, штрафы; ручное изменение уровня. |
| 5.6 | Ручные начисления/списания | Форма: пользователь, сумма, тип, комментарий; массовое начисление (CSV или фильтр). |
| 5.7 | Штрафы | Просмотр списка штрафов пользователя, снятие балла с указанием причины, пересчёт уровня. |
| 5.8 | Заявки на обмен | Список заявок, фильтр по статусу, отметка о выдаче/отмене. |
| 5.9 | Аудит | Лог изменений (кто, когда, что): уровни, транзакции, квесты, товары. |
| 5.10 | Уведомления и аналитика | Рассылки пользователям/группам; сводки по балансам, оборотам, квестам, товарам (по приоритету). |

### Этап 6: Интеграция с основной системой

| № | Задача | Детали |
|---|--------|--------|
| 6.1 | Спецификация API основной системы | Документирование эндпоинтов и форматов: профиль, смены, отмены, no-show (см. ARCHITECTURE.md). |
| 6.2 | Реализация клиента | Вызовы из нашего бэкенда к API основной системы (авторизация, смены, события). |
| 6.3 | Webhook или поллинг | Приём событий «смена завершена», «отмена», «no-show» или периодический опрос; идемпотентность обработки. |

### Этап 7: Дополнительно (по приоритету)

| № | Задача | Детали |
|---|--------|--------|
| 7.1 | Уведомления в дашборде | Экран/список уведомлений по клику на колокольчик; пометка прочитано. |
| 7.2 | Push-уведомления | Опционально: подписка, отправка при новом квесте, уровне, начислении. |
| 7.3 | Аналитика (события) | События на фронте (просмотр экранов, клики по квестам/магазину) для отчётов и оптимизации. |
| 7.4 | Тесты | Unit/интеграционные тесты для критичных расчётов (уровень, штрафы, начисление за квест). |
