## Что нужно сделать

### Принцип: сначала наш бэкенд и моки, потом внешний API

- **Наши данные** (баланс, уровни, квесты, магазин, транзакции, штрафы, заявки) — сразу хранятся в нашей БД и отдаются нашим API.
- **Данные основной системы** (смены, отмены, прогулы, профиль пользователя, авторизация) — до подключения реального API используем **плейсхолдеры/моки**:
  - тестовые пользователи и «заглушка» авторизации;
  - ручной ввод или админские действия вместо событий «смена завершена» / «отмена» / «no-show»;
  - при необходимости — мок-сервис или конфиг с тестовыми сменами.
- **Этап интеграции с API основной системы** — в конце: замена заглушек на реальные вызовы и приём событий.

---

### Этап 1: Наш бэкенд и база данных

Без обращения к внешнему API. Все данные — в нашей БД.

| № | Задача | Детали |
|---|--------|--------|
| 1.1 | Стек бэкенда | Next.js API Routes или отдельный сервис (Node/Python/Go). |
| 1.2 | База данных | СУБД (например PostgreSQL), схема по ARCHITECTURE.md: users, levels, strikes, transactions, store_items, redemptions, quests, quest_progress, audit_log. |
| 1.3 | Миграции | Инструмент миграций, первичное заполнение справочника уровней. |
| 1.4 | API для дашборда | Эндпоинты: текущий пользователь (профиль, баланс, уровень, штрафы за 30 дней), история транзакций, квесты с прогрессом, каталог магазина, создание заявки на обмен. |

**Плейсхолдер:** «текущий пользователь» отдаётся по идентификатору из сессии/токена; до этапа авторизации можно использовать фиксированный test user id из конфига или query-параметр для разработки.

---

### Этап 2: Авторизация (плейсхолдер)

Логин и сессия без интеграции с основной системой.

| № | Задача | Детали |
|---|--------|--------|
| 2.1 | Простая авторизация | Вход по логину/паролю или «dev-режим» (фиксированный пользователь из БД). Создание сессии (JWT или cookie). |
| 2.2 | Сессии на фронте | Передача токена/cookie в запросах к нашему API. |
| 2.3 | Первый вход | При первом «логине» создаётся или обновляется запись пользователя в нашей БД (пока без external_id или с заглушкой). |

**Плейсхолдер:** профиль (имя, аватар) храним в нашей БД или подставляем тестовые значения. Позже (этап интеграции) заменим на получение из API основной системы и маппинг по external_id.

---

### Этап 3: Подключение фронта к нашему API

Фронт ходит только в наш бэкенд. Данные пользователя, баланс, история, квесты, магазин — из нашей БД.

| № | Задача | Детали |
|---|--------|--------|
| 3.1 | Замена моков на вызовы API | В `page.tsx` (и при необходимости в других компонентах) убрать константы USER, EARNINGS, QUESTS, STORE_ITEMS; получать данные с наших эндпоинтов (fetch / React Query / SWR). |
| 3.2 | Состояния загрузки и ошибок | Скелетоны, спиннеры, сообщения об ошибке при недоступности API. |

Данные по сменам (shifts_completed) пока берём из нашей БД (поля пользователя или отдельной таблицы), обновляемые вручную или через админку (см. этап 4).

---

### Этап 4: Механики на наших данных и моках событий

Вся логика уровней, квестов и штрафов работает в нашем бэкенде. События «смена завершена», «отмена», «no-show» **не** приходят из внешнего API — имитируем их.

| № | Задача | Детали |
|---|--------|--------|
| 4.1 | Начисление монет за смену | Эндпоинт или внутренний сервис: «засчитать завершённую смену» для пользователя (user_id, сумма монет, описание). Обновление баланса, транзакция, увеличение shifts_completed. **Мок:** вызов вручную (админ или dev-инструмент) или тестовый скрипт. |
| 4.2 | Пересчёт уровня | При изменении shifts_completed пересчитывать уровень пользователя по справочнику уровней. |
| 4.3 | Квесты | Логика условий (смены за день/неделю, серия, ранняя смена, рейтинг). Прогресс хранить в quest_progress; при выполнении — начисление монет и транзакция. **Мок:** данные для проверки условий (число смен, время смены, рейтинг) пока вводим вручную или через админку/сидеры, без API основной системы. |
| 4.4 | Штрафы и понижение уровня | Эндпоинт или внутренний сервис: «зарегистрировать штраф» (user_id, тип: late_cancel / no_show). Запись в strikes, подсчёт за 30 дней, при достижении порога уровня — понижение. **Мок:** вызов вручную (админ или dev) вместо реальных событий от основной системы. |
| 4.5 | Отображение штрафов в UI | Показ «N/M штрафов (до понижения)» на главной или в блоке уровня (данные с нашего API). |

После этапа интеграции с API основной системы вызовы «засчитать смену» и «зарегистрировать штраф» будут вызываться автоматически при получении событий из внешней системы.

### Этап 5: Магазин и обмен

Только наш бэкенд и наша БД.

| № | Задача | Детали |
|---|--------|--------|
| 5.1 | Списание и заявка | При «Купить» на фронте: запрос к нашему API → проверка баланса, списание монет, создание транзакции и заявки (status: pending). |
| 5.2 | Обработка заявок | Для админа: список заявок, смена статуса (fulfilled / cancelled), при отмене — возврат монет (опционально). Выдача награды — ручная или отдельная интеграция позже. |

---

### Этап 6: Админ-панель

Роут `/admin`, все данные из нашей БД. Никаких вызовов к основной системе на этом этапе.

| № | Задача | Детали |
|---|--------|--------|
| 6.1 | Роут и доступ | Страницы под `/admin`, проверка роли admin (пока можно по нашему полю в БД или конфигу). |
| 6.2 | Управление магазином | CRUD товаров: название, описание, категория, цена, лимит, даты, видимость, порядок. |
| 6.3 | Управление квестами | CRUD квестов: период, условие, награда, привязка к группам. |
| 6.4 | Справочник уровней | Редактирование порога смен, порога штрафов, перков. |
| 6.5 | Пользователи | Поиск по ID/email/имени, карточка: баланс, уровень, смены, штрафы; ручное изменение уровня. |
| 6.6 | Ручные начисления/списания | Форма: пользователь, сумма, тип, комментарий; при необходимости — массовое начисление (CSV или фильтр). |
| 6.7 | Штрафы | Просмотр штрафов пользователя, снятие балла с причиной, пересчёт уровня. |
| 6.8 | Заявки на обмен | Список заявок, фильтр по статусу, отметка о выдаче/отмене. |
| 6.9 | Аудит | Лог изменений (кто, когда, что). |
| 6.10 | Имитация событий смен (мок) | В админке или отдельном dev-инструменте: кнопки/формы «Засчитать смену», «Зарегистрировать позднюю отмену», «Зарегистрировать прогул» для выбранного пользователя — вызывают ту же логику, что потом будет вызываться из webhook/API основной системы. |

---

### Этап 7: Интеграция с API основной системы (последний)

Замена плейсхолдеров и моков на реальные вызовы и события.

| № | Задача | Детали |
|---|--------|--------|
| 7.1 | Спецификация API основной системы | Документирование эндпоинтов и форматов: авторизация (токен/SSO), профиль пользователя, смены, отмены, no-show (см. ARCHITECTURE.md). |
| 7.2 | Клиент API основной системы | Реализация вызовов из нашего бэкенда: проверка токена, получение профиля по external_id. |
| 7.3 | Авторизация через основную систему | Замена простого логина на OAuth/SSO основной платформы; при первом входе создание/обновление пользователя с external_id и при необходимости обновление имени/аватара из их API. |
| 7.4 | События смен и штрафов | Подключение webhook или поллинга: при событии «смена завершена» — вызов нашей логики начисления за смену; при «отмена &lt;24 ч» и «no-show» — вызов нашей логики регистрации штрафа. Идемпотентность по external_id смены/бронирования. |
| 7.5 | Данные для квестов | Получение из API основной системы списка/фактов смен (за период, с датой/временем, рейтингом) для расчёта условий квестов; замена ручного ввода/моков на вызовы к их API при пересчёте прогресса. |
| 7.6 | Удаление или отключение моков | Отключение dev-логина и тестовых пользователей; удаление или флаг для админских «кнопок» имитации событий, если они больше не нужны в проде. |
