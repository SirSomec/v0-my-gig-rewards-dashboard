# Архитектура проекта MyGig Rewards Dashboard

Описание компонентов системы, разграничение данных между собственной базой и основной платформой, интеграция по API.

---

## 1. Общая схема

```
┌─────────────────────────────────────────────────────────────────────────┐
│  Пользователь (гиг-работник) / Администратор                            │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────┐
│  ФРОНТЕНД (Next.js)                                                      │
│  — Дашборд наград (баланс, уровень, квесты, магазин, история)           │
│  — Админ-панель (/admin)                                                 │
└─────────────────────────────────────────────────────────────────────────┘
                                    │
                    ┌───────────────┴───────────────┐
                    ▼                               ▼
┌───────────────────────────────┐   ┌───────────────────────────────────┐
│  НАШ БЭКЕНД (API + БД)         │   │  ОСНОВНАЯ СИСТЕМА (Core Platform)  │
│  — Авторизация сессии (JWT)    │   │  — Учёт пользователей, смен,        │
│  — Баланс, уровни, квесты      │◄──┤    бронирований, отмен, прогулы     │
│  — Магазин, начисления, штрафы │   │  — Источник правды по сменам       │
│  — Собственная БД              │   │  — Внешнее API для интеграции      │
└───────────────────────────────┘   └───────────────────────────────────┘
```

- **Фронтенд** — текущее Next.js-приложение; обращается к нашему бэкенду (и при необходимости к основной системе через наш бэкенд или прокси).
- **Наш бэкенд** — сервис наград: своя БД, свои API, бизнес-логика монет, уровней, квестов, магазина. При необходимости запрашивает данные из основной системы по API.
- **Основная система** — платформа гиг-работы (бронирование смен, учёт явок, отмен, рейтинги). Является источником правды по сменам и событиям; награды мы считаем и храним у себя.

---

## 1.1 Бэкенд: шаблон микросервиса (обязательно)

Разработка бэкенда ведётся **на базе принятого в команде шаблона микросервиса** в папке репозитория:

**`nestjs-service/`** — шаблон микросервиса на NestJS.

Использование шаблона обязательно: единая структура проекта, стек, деплой и инструменты (миграции, health, логи, Helm) упрощают поддержку и интеграцию.

**Что входит в шаблон:**

| Компонент | Описание |
|-----------|----------|
| **Стек** | NestJS 11, TypeScript, Drizzle ORM, PostgreSQL, Redis, опционально Mongoose (MongoDB). |
| **Инфраструктура** | Модули БД (Drizzle), Redis, кэш; валидация конфигурации (Joi); healthcheck (`@mygigtechnologies/healthcheck`), логирование (`@mygigtechnologies/nest-logger`). |
| **API** | Swagger (документация), версионирование по URI (v1). |
| **Авторизация** | Passport, Basic Auth (стратегия в шаблоне); при необходимости дополняется JWT/OAuth для интеграции с основной системой. |
| **Миграции** | Drizzle Kit: генерация и применение миграций (`db:generate`, `db:migrate`). |
| **Деплой** | Helm-чарт для Kubernetes (в т.ч. окружения nonproduction/production). |
| **Тесты** | Jest, e2e, Pact (contract tests), Testcontainers для PostgreSQL. |

**Инициализация нового сервиса из шаблона:**

1. Скопировать или клонировать содержимое `nestjs-service/` в каталог будущего сервиса (например `rewards-api/` или оставить в монорепе).
2. Установить зависимости: `npm i`.
3. Запустить скрипт инициализации (замена плейсхолдеров на имя сервиса):
   ```bash
   ./init.sh --name "MyGig Rewards API" --description "Сервис наград и лояльности"
   ```
   Плейсхолдеры `<SERVICE_NAME>`, `<SERVICE_SLUG>`, `<SERVICE_DESCRIPTION>` будут заменены во всех файлах.
4. Добавить модули, контроллеры и схемы БД под домен наград (пользователи, уровни, транзакции, квесты, магазин и т.д.) по ARCHITECTURE.md.

Подробности по структуре и скриптам — в **`nestjs-service/README.md`**.

## 2. Собственная база данных (наши данные)

В нашей БД храним только то, что относится к программе лояльности и наград. Пользователей и смены не дублируем как источник правды — только ссылки на идентификаторы основной системы.

**Реализация:** схемы Drizzle в `nestjs-service/src/infra/db/drizzle/schemas/` (levels, users, strikes, transactions, store_items, redemptions, quests, quest_progress, audit_log). API дашборда — модуль `rewards` (`GET/POST /v1/rewards/...`).

### 2.1 Пользователи и профиль наград

| Поле / сущность | Описание |
|-----------------|----------|
| `user_id` (PK) | Внутренний ID (или ID из основной системы, если он стабильный и уникальный). |
| `external_id` | ID пользователя в основной системе (обязательная связь). |
| `balance` | Текущий баланс Gig Coins (целое число). |
| `level_id` | FK на справочник уровней. |
| `shifts_completed` | Количество завершённых смен (можно считать у себя по событиям из API или синхронизировать с основной системой). |
| `name`, `email`, `avatar_url` | Кэш для отображения в дашборде; обновляется при логине или по API основной системы (опционально). |
| `created_at`, `updated_at` | Служебные поля. |

При первом входе (по токену/SSO основной системы) создаём запись пользователя с `external_id` и стартовым уровнем (Бронза).

### 2.2 Справочник уровней

| Поле | Описание |
|------|----------|
| `id` (PK) | Внутренний ID. |
| `name` | Название (Бронзовый новичок, Серебряный партнёр и т.д.). |
| `shifts_required` | Порог завершённых смен для перехода на этот уровень. |
| `strike_threshold` | Порог штрафов за 30 дней, при достижении которого уровень понижается (NULL для Бронзы). |
| `perks` | JSON или отдельная таблица: список перков (название, описание). |
| `sort_order` | Порядок отображения. |

Данные полностью наши; админ настраивает уровни и пороги (см. FUNCTIONALITY.md).

### 2.3 Штрафные баллы (strikes)

| Поле | Описание |
|------|----------|
| `id` (PK) | Внутренний ID. |
| `user_id` | FK на пользователя. |
| `type` | `no_show` \| `late_cancel`. |
| `shift_external_id` | ID смены в основной системе (для аудита). |
| `occurred_at` | Дата/время события (для окна 30 дней). |
| `created_at` | Когда запись создана у нас (например, по вебхуку или джобе). |

По этим записям считаем число штрафов за последние 30 дней и сравниваем с `strike_threshold` текущего уровня пользователя.

### 2.4 Начисления и списания монет (транзакции)

| Поле | Описание |
|------|----------|
| `id` (PK) | Внутренний ID. |
| `user_id` | FK на пользователя. |
| `amount` | Сумма (+ начисление, − списание). |
| `type` | `shift` \| `bonus` \| `quest` \| `manual_credit` \| `manual_debit` \| `redemption` и т.д. |
| `source_ref` | Внешняя ссылка (ID смены, ID квеста, ID заявки на обмен) — опционально. |
| `title`, `description` | Для отображения в истории (например, «Смена на складе», «Бонус за выходные»). |
| `location` | Опционально (локация смены). |
| `client_id` | ID или код бренда/клиента (для квестов по сменам или часам в клиенте). |
| `category` | Категория/профессия смены (для квестов по категории). |
| `hours` | Отработанные часы в смене (для квестов по часам). |
| `created_at` | Время операции. |
| `created_by` | Для ручных операций — ID админа (опционально). |

Баланс пользователя обновляется при каждой транзакции. При записи смены (админка или API) можно передать `client_id`, `category` и `hours` — они сохраняются в транзакции и используются при расчёте прогресса квестов (условия по клиенту, категории, часам).

### 2.5 Магазин (товары)

| Поле | Описание |
|------|----------|
| `id` (PK) | Внутренний ID. |
| `name`, `description` | Название и описание. |
| `category` | Скидка, бустер, мерч, подарок и т.д. |
| `cost` | Цена в Gig Coins. |
| `icon` | Код иконки (discount, booster, merch, gift). |
| `stock_limit` | NULL = без лимита, иначе — **общий лимит тиража** (максимальное число выкупов этого товара всеми пользователями). Остаток = stock_limit − число заявок (redemptions) по этому товару со статусами pending и fulfilled. При создании заявки проверяется остаток; при 0 — ошибка «Товар закончился». |
| `visible_from`, `visible_until` | Период показа. |
| `is_active` | Вкл/выкл без удаления. |
| `sort_order` | Порядок в каталоге. |
| `visibility_rules` | JSON (опционально): уровень, регион, тег — для кого показывать. |

Всё управляется админом; данные только в нашей БД. API каталога (`GET /v1/rewards/store`) возвращает по каждому товару также `stockLimit` и `redeemedCount` для отображения остатка на фронте.

### 2.6 Заявки на обмен (redemptions)

| Поле | Описание |
|------|----------|
| `id` (PK) | Внутренний ID. |
| `user_id` | FK на пользователя. |
| `store_item_id` | FK на товар. |
| `status` | `pending` \| `fulfilled` \| `cancelled`. |
| `coins_spent` | Сумма списания (на момент заказа). |
| `created_at` | Время заявки. |
| `processed_at`, `processed_by` | Когда и кем обработано (для админа). |
| `notes` | Комментарий при выдаче/отмене. |

Списание монет создаёт транзакцию с `type = redemption` и ссылкой на `id` заявки.

### 2.7 Квесты (шаблоны)

| Поле | Описание |
|------|----------|
| `id` (PK) | Внутренний ID. |
| `name`, `description` | Название и описание цели. |
| `period` | `daily` \| `weekly` \| `monthly`. |
| `condition_type` | Тип условия: `shifts_count`, `shifts_count_client`, `shifts_count_clients`, `shifts_count_category`, `hours_count`, `hours_count_client`, `hours_count_clients`. |
| `condition_config` | JSON параметров: `total` (число смен), `totalHours` (часы), `clientId`, `clientIds[]`, `category` — в зависимости от типа. |
| `reward_coins` | Награда в монетах. |
| `icon` | streak, target, calendar, trophy. |
| `is_active` | Вкл/выкл. |
| `is_one_time` | Единоразовый квест (пользователь выполняет один раз). |
| `active_from`, `active_until` | Окно активности (опционально); после `active_until` квест не показывается (автоотключение по периоду). |
| `target_type` | `all` \| `group` — для всех или для групп. |
| `target_group_id` | FK на группу (если target_type = group); NULL для глобальных. |

Группы пользователей (для квестов и видимости товаров) можно хранить в отдельной таблице с правилами (уровень, регион, партнёр, тег) или связью user_id ↔ group_id.

### 2.8 Прогресс по квестам

| Поле | Описание |
|------|----------|
| `id` (PK) | Внутренний ID. |
| `user_id` | FK на пользователя. |
| `quest_id` | FK на шаблон квеста. |
| `period_key` | Идентификатор периода: дата (YYYY-MM-DD) для daily, дата понедельника для weekly, YYYY-MM для monthly; для единоразового квеста — `'once'`. |
| `progress` | Текущее значение (например, число завершённых смен за период). |
| `completed_at` | NULL или время выполнения (тогда начисление уже выдано). |
| `created_at`, `updated_at` | Служебные поля. |

Уникальность: один `(user_id, quest_id, period_key)`. Начисление монет за квест создаём один раз при `completed_at` и пишем транзакцию.

### 2.9 Админка и аудит

| Сущность | Назначение |
|----------|------------|
| **admin_users** | Логин/роль админа (или переиспользуем основную систему с ролью admin). |
| **audit_log** | Кто (admin_id), когда, какое действие (изменение уровня, ручное начисление, снятие штрафа, изменение квеста/товара). Опционально: старые/новые значения. |

Остальное (уведомления, шаблоны рассылок) при необходимости тоже храним в своей БД.

## 3. Данные из основной системы (получение по API)

Эти данные не храним как источник правды у себя; получаем по API основной платформы для расчёта наград, штрафов и отображения в дашборде.

### 3.1 Авторизация и пользователь

| Что нужно | Зачем |
|-----------|--------|
| Валидация токена / SSO (JWT или OAuth2) | Понимать, кто зашёл в дашборд; выдать или создать запись в нашей БД по `external_id`. |
| Профиль пользователя (id, name, email, avatar_url) | Отображение в шапке; кэш в нашей БД по желанию. |

Рекомендуется: единый вход через основную систему (редирект на неё, затем callback с токеном); наш бэкенд по токену получает `external_id` и профиль.

### 3.2 Смены

| Что нужно | Зачем |
|-----------|--------|
| Список смен пользователя за период (или факт завершения) | Подсчёт `shifts_completed`, прогресс по уровню, выполнение квестов («5 смен за неделю», «3 смены подряд»). |
| По одной смене: id, start_time, end_time, status, location, partner/site_id, rating (если есть) | Определение типа начисления (смена/бонус), локация и название в истории; квесты «ранняя пташка» (start_time), «рейтинг 5★ на N сменах». |
| События: «смена завершена» (completed) | Триггер начисления монет за смену и пересчёт прогресса квестов. |

Формат: REST (список смен за период, детали смены) и/или webhook при смене статуса смены на «завершена».

### 3.3 Бронирования и отмены

| Что нужно | Зачем |
|-----------|--------|
| Факт отмены бронирования + время отмены и время начала смены | Определить «поздняя отмена» (менее 24 ч до старта) → начисление штрафного балла в нашей БД. |
| Список отмен за период (user_id, shift_id, cancelled_at, shift_start_at) | Реакция на событие отмены (вебхук или опрос API) и аудит. |

Желательно получать событие «бронирование отменено» с полями `cancelled_at` и `shift_start_at`, чтобы не опрашивать список смен постоянно.

### 3.4 Прогулы (no-show)

| Что нужно | Зачем |
|-----------|--------|
| Факт no-show по смене (user_id, shift_id, дата) | Начисление штрафного балла типа `no_show` в нашей БД. |

Обычно приходит как событие из основной системы (смена в статусе no-show или отдельное событие «работник не явился»).

### 3.5 Опционально: справочники

| Что нужно | Зачем |
|-----------|--------|
| Регионы, партнёры (склады, площадки) | Группы для квестов и видимости товаров («только для региона X», «только партнёр Y»). |

Можно получать по API при настройке админом или кэшировать у себя с периодическим обновлением.

## 4. Потоки данных (кратко)

1. **Вход пользователя**  
   Основная система выдаёт токен → наш бэкенд проверяет токен, получает/обновляет профиль по API, создаёт/обновляет запись пользователя и уровень в своей БД → фронт получает от нашего API баланс, уровень, квесты, историю.

2. **Завершение смены**  
   Основная система шлёт событие (webhook) или наш сервис периодически опрашивает API → наш бэкенд начисляет монеты, обновляет `shifts_completed`, пересчитывает уровень и прогресс квестов → при выполнении квеста — ещё одна транзакция и обновление `quest_progress`.

3. **Отмена менее чем за 24 ч**  
   Основная система шлёт событие отмены → наш бэкенд проверяет `cancelled_at` и `shift_start_at`, при необходимости создаёт strike `late_cancel`, пересчитывает штрафы за 30 дней и при достижении порога понижает уровень.

4. **Прогул**  
   Основная система шлёт событие no-show → наш бэкенд создаёт strike `no_show`, пересчитывает штрафы и уровень.

5. **Обмен в магазине**  
   Пользователь на фронте нажимает «Купить» → запрос к нашему API → проверка баланса, списание монет, создание заявки на обмен и транзакции — всё в нашей БД. Выдача награды (мерч, код скидки и т.д.) — ручная или интеграция с партнёрами вне этой схемы.

6. **Админ**  
   Все операции админа (магазин, квесты, уровни, ручные начисления/списания, снятие штрафов) идут в наш бэкенд и нашу БД; при необходимости админ смотрит данные основной системы через наш бэкенд (например, список смен пользователя), но не меняет их там.

---

## 5. Резюме: что где хранится

| Данные | Где хранятся | Примечание |
|--------|--------------|------------|
| Баланс монет, уровень, штрафы, транзакции, квесты (шаблоны и прогресс), магазин, заявки на обмен | Наша БД | Единственный источник правды по наградам. |
| Справочник уровней (название, пороги смен и штрафов, перки) | Наша БД | Настраивается админом. |
| Идентификация пользователя (external_id), кэш профиля (имя, аватар) | Наша БД | Профиль можно подтягивать из основной системы при логине. |
| Факты смен (завершены, отмены, no-show), рейтинги, расписание | Основная система | Получаем по API / webhook для расчёта наград и штрафов. |
| Авторизация (выдача токена, SSO) | Основная система (или мы по её OAuth) | Наш бэкенд только проверяет токен и маппит на user_id. |

Документ можно дополнять конкретными форматами API (REST/JSON, webhook payloads) по мере согласования с командой основной платформы. См. также FUNCTIONALITY.md и DEPLOY.md.
