# Логика запросов к TOJ: тайминги, ограничения, примеры

Описание текущей реализации обращений к TOJ из бэкенда Rewards: когда и какие запросы выполняются, используемые ограничения и примеры тел запросов.

---

## 1. Общая логика

К TOJ обращается только **синхронизация смен** (модуль `toj`, сервисы `TojClientService` и `TojSyncService`). Других исходящих вызовов к TOJ в проекте нет.

- **Единственный используемый эндпоинт TOJ:** `POST /job.find-many.query` (список смен с фильтрами и пагинацией).
- **Когда вызывается:** при ручном запуске синхронизации (`POST /v1/admin/toj-sync/run`), при загрузке главной страницы дашборда (вызов `GET /v1/rewards/me` — перед ответом вызывается `runSyncIfNeeded(5 мин)`), или при будущем запуске по cron. При загрузке главной синхронизация выполняется **не чаще чем раз в 5 минут** (время последнего запуска хранится в `system_settings`, ключ `toj_sync_last_run_at`).
- **Условия входа:** должны быть заданы `TOJ_BASE_URL`, `TOJ_USER`, `TOJ_PASSWORD` и включена синхронизация (`TOJ_SYNC_ENABLED=true` или `1`). Иначе запросы к TOJ не выполняются.

## 2. Тайминги и водяной знак (watermark)

| Параметр | Переменная окружения | Значение по умолчанию | Назначение |
|----------|----------------------|------------------------|------------|
| Водяной знак | — | хранится в БД | В `system_settings` ключ `toj_sync_last_updated_at`. Значение — ISO-строка даты. Запросы к TOJ передают только смены с `updatedAt >= watermark`. |
| Начальный водяной знак (первый запуск) | `TOJ_SYNC_INITIAL_DAYS_AGO` | `7` | Если watermark в БД нет, он задаётся как «сейчас минус N дней». Так ограничивается глубина первого прогона. |

**Тайминги запросов:**

- Запросы к TOJ выполняются **последовательно** в рамках одного прогона: батч workerIds → цикл страниц (limit/skip), затем следующий батч.
- **Бизнес-правило:** смена может изменить свой статус в течение **30 дней после даты старта** (например, confirmed → cancelled). Мы фильтруем по `updatedAt`, а не по дате смены: при смене статуса TOJ обновляет запись смены, поэтому следующий прогон синхронизации подхватит изменение. Не следует добавлять фильтр «только смены за последние N дней по дате старта» — иначе такие поздние смены статуса будут пропущены.
- Фильтр по статусу **не передаётся** — загружаются смены со всеми статусами. Для каждой смены: **confirmed** → начисление (если ещё нет транзакции с таким source_ref); **cancelled** → при initiatorType/initiator = worker и отмена менее чем за 24 ч до начала — штраф «поздняя отмена»; остальные статусы — пропуск (причина wrongStatus).
- Таймаутов и повторных попыток на уровне кода не задано — используется поведение `fetch` по умолчанию.
- После успешного прогона watermark обновляется максимальным `updatedAt` среди обработанных смен (или остаётся прежним, если ничего не обработано).

## 3. Установленные ограничения

Все ограничения задаются переменными окружения (опционально), при отсутствии — используются значения по умолчанию в коде.

| Ограничение | Переменная | По умолчанию | Описание |
|-------------|------------|--------------|----------|
| Максимум смен за один прогон | `TOJ_SYNC_MAX_JOBS_PER_RUN` | `1000` | Суммарно обрабатываем + пропускаем не более этого числа смен за один вызов `runSync()`. Достигнув лимита, цикл по батчам и страницам прерывается. |
| Размер страницы в запросе к TOJ | `TOJ_SYNC_PAGE_SIZE` | `500` | Параметр `limit` в теле `job.find-many.query`. В `TojClientService` дополнительно ограничен сверху значением 1000. |
| Размер батча workerIds | `TOJ_SYNC_WORKER_BATCH_SIZE` | `200` | Сколько `external_id` пользователей передаётся в одном запросе в фильтре `workerIds`. Список всех пользователей с `external_id` разбивается на батчи по этому размеру. |

**Ограничения внутри TojClientService (жёстко в коде):**

- `limit`: минимум 1, максимум 1000; итоговое значение = `min(max(option.limit, 1), 1000)`.
- `skip`: не меньше 0.

## 4. Порядок выполнения одного прогона синхронизации

1. Проверка: TOJ настроен и синхронизация включена; иначе возврат без запросов к TOJ.
2. Чтение watermark из `system_settings`; если пусто — watermark = сейчас − `TOJ_SYNC_INITIAL_DAYS_AGO` дней.
3. Выборка из БД: пользователи с непустым `external_id`; формирование списка `workerIds` и маппинга `workerId → { id, createdAt }`.
4. Цикл по батчам `workerIds` (размер батча = `TOJ_SYNC_WORKER_BATCH_SIZE`):
   - Внутренний цикл по страницам (`skip = 0`, затем `skip += pageSize`):
     - Если уже обработано + пропущено + штрафов поздняя отмена ≥ `TOJ_SYNC_MAX_JOBS_PER_RUN` — выход из циклов.
     - Запрос к TOJ: `POST /job.find-many.query` с фильтрами (текущий батч `workerIds`, `updatedAt >= watermark`); **фильтр по статусу не передаётся** — загружаются смены со всеми статусами.
     - Для каждой смены в ответе: поиск user по `workerId`; по статусу смены: **confirmed** — проверки (дата смены ≥ created_at пользователя, нет транзакции с таким `source_ref`), при успехе начисление; **cancelled** — при initiatorType/initiator = worker и отмена < 24 ч до начала вызов `processLateCancelIfEligible` (штраф «поздняя отмена»); остальные статусы — пропуск (wrongStatus). Обновление `maxUpdatedAt` по полю смены.
     - Если получено смен меньше, чем `pageSize`, выход из внутреннего цикла; иначе `skip += items.length` и следующая страница.
5. Если `maxUpdatedAt` изменился — запись нового watermark в `system_settings`.
6. Возврат результата: `{ processed, skipped, lateCancelApplied?, skippedReasons?, errors, watermark }`.

## 5. Примеры генерируемых запросов к TOJ

Все запросы — **POST** на `{TOJ_BASE_URL}/job.find-many.query`, заголовки: `Content-Type: application/json`, `Authorization: Basic <base64(TOJ_USER:TOJ_PASSWORD)>`.

### 5.1 Первая страница первого батча (например, 200 workerIds)

```json
{
  "data": {
    "filters": {
      "workerIds": ["ext-1", "ext-2", "..."],
      "updatedAt": ["gte:2025-02-28T00:00:00.000Z"]
    },
    "projection": "",
    "options": {
      "limit": 500,
      "skip": 0,
      "sort": { "updatedAt": -1 }
    }
  }
}
```

- `workerIds` — до 200 значений (или сколько есть в первом батче).
- `updatedAt` — текущее значение watermark (в первом запуске — дата «сейчас − TOJ_SYNC_INITIAL_DAYS_AGO»).
- `limit` — значение `TOJ_SYNC_PAGE_SIZE` (по умолчанию 500).

### 5.2 Вторая страница того же батча

```json
{
  "data": {
    "filters": {
      "workerIds": ["ext-1", "ext-2", "..."],
      "updatedAt": ["gte:2025-02-28T00:00:00.000Z"]
    },
    "projection": "",
    "options": {
      "limit": 500,
      "skip": 500,
      "sort": { "updatedAt": -1 }
    }
  }
}
```

Отличие от 5.1 — только `skip: 500`. Дальнейшие страницы: `skip: 1000`, `1500`, …

### 5.3 Первая страница второго батча workerIds

Те же структура и правила, но в `filters.workerIds` — следующие 200 `external_id`. Фильтры `statuses` и `updatedAt` те же, `skip` снова 0 для первой страницы батча.

### 5.4 Минимальный по объёму запрос (один workerId, маленькая страница)

Если бы батч был из одного workerId и `TOJ_SYNC_PAGE_SIZE=10`:

```json
{
  "data": {
    "filters": {
      "workerIds": ["single-worker-id"],
      "statuses": ["confirmed"],
      "updatedAt": ["gte:2025-03-01T12:00:00.000Z"]
    },
    "projection": "",
    "options": {
      "limit": 10,
      "skip": 0,
      "sort": { "updatedAt": -1 }
    }
  }
}
```

## 6. Сводка переменных окружения (TOJ)

| Переменная | Назначение |
|------------|------------|
| `TOJ_BASE_URL` | Базовый URL TOJ (без завершающего слэша). Обязателен для запросов. |
| `TOJ_USER` | Логин Basic Auth. |
| `TOJ_PASSWORD` | Пароль Basic Auth. |
| `TOJ_SYNC_ENABLED` | Включение синхронизации: `true` или `1`. |
| `TOJ_SYNC_MAX_JOBS_PER_RUN` | Макс. смен за один прогон (по умолчанию 1000). |
| `TOJ_SYNC_INITIAL_DAYS_AGO` | Дней «назад» для начального watermark (по умолчанию 7). |
| `TOJ_SYNC_WORKER_BATCH_SIZE` | Размер батча workerIds в одном запросе (по умолчанию 200). |
| `TOJ_SYNC_PAGE_SIZE` | Размер страницы в запросе к TOJ (по умолчанию 500). |

Спецификация API TOJ (эндпоинты, форматы) описана в [TOJ-API.md](./TOJ-API.md). План синхронизации и идемпотентность — в [TOJ-SYNC-PLAN.md](./TOJ-SYNC-PLAN.md).
