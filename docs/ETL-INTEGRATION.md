# Поэтапная интеграция ETL в проект MyGig Rewards

Данные ETL (ClickHouse, база `etl`: таблицы `mygig_documents`, `mygig_job_history`, `mg_clients`) уже доступны для просмотра в админке («Данные ETL»). Ниже — план использования этих данных **непосредственно в логике приложения**: начисления за смены, квесты, справочники, отчёты.

---

## Текущее состояние

| Компонент | Статус |
|-----------|--------|
| Подключение к ETL (ClickHouse 8443) | ✅ EtlExplorerService, чтение в админке |
| Наша БД (PostgreSQL) | ✅ users, transactions, strikes, quests, store_items, redemptions |
| Начисление за смену | ✅ `POST /v1/rewards/shifts/complete` (ручной ввод / мок в админке) |
| Штрафы | ✅ `POST /v1/rewards/strikes` (ручной ввод / мок) |
| Источник смен/штрафов в проде | ⏳ Планируется: API основной системы (ROADMAP этап 7) **или ETL** |

ETL можно использовать как **альтернативный или дополнительный** источник фактов смен и справочников до или вместо подключения API основной системы.

---

## Цели интеграции

1. **Смены из ETL** — брать завершённые смены из `mygig_job_history` (или аналога), маппить на пользователя по `external_id`, вызывать существующую логику начисления (`recordShiftCompleted`) с идемпотентностью по ID смены.
2. **Справочник клиентов** — использовать `mg_clients` для подстановки названий клиентов в истории/квестах и в админке (без дублирования в нашей БД или с лёгкой синхронизацией).
3. **Документы (опционально)** — при необходимости использовать `mygig_documents` для отображения или проверок (например, статус договора при первом входе).

---

## Фаза 0: Спецификация и маппинг (подготовка)

**Задачи:**

1. **Описать схему ETL-таблиц**  
   Зафиксировать в репозитории (например, `docs/ETL-SCHEMA.md`): список таблиц, ключевые колонки `mygig_job_history`, `mg_clients`, `mygig_documents` — идентификаторы, даты, статусы, ссылки на пользователя/клиента. Данные взять из ETL Explorer (колонки, превью).

2. **Определить маппинг идентификаторов**  
   - Как в ETL однозначно идентифицировать пользователя (поле для связи с `users.external_id`)?  
   - Какой столбец в `mygig_job_history` считать ID смены для `source_ref` и идемпотентности?  
   - Как связать смену с клиентом для квестов (`client_id` / `mg_clients`)?

3. **Решить формат «завершённой смены»**  
   Какой статус/тип записи в ETL считаем «сменой к начислению» (например, статус `completed` или аналог). При необходимости — фильтр по дате, чтобы не обрабатывать старые данные дважды.

**Результат:** документ с маппингом ETL → наша модель (user, shift_id, client_id, дата, часы, категория и т.д.) и критерий «новая смена к обработке».

---

## Фаза 1: Только чтение ETL в приложении (без записи в нашу БД)

**Цель:** использовать ETL как источник данных для отображения и отладки, не меняя существующие потоки начисления.

**Задачи:**

1. **Сервис доступа к ETL (поверх EtlExplorerService)**  
   Выделить слой, например `EtlDataService`, с методами типа:
   - `getRecentCompletedJobs(userExternalId?, limit, since?)` — выборка из `mygig_job_history` по пользователю и дате;
   - `getClientById(clientId)` / `getClients()` — чтение из `mg_clients`.
   Запросы только SELECT, без записи в PostgreSQL.

2. **Эндпоинты для админки (опционально)**  
   Например:
   - `GET /v1/admin/etl/jobs?userId=&since=&limit=` — последние смены из ETL для выбранного пользователя (по `external_id`).  
   Использование: сравнение «что в ETL» и «что уже засчитано у нас» (транзакции с `source_ref`).

3. **Отображение в админке**  
   В карточке пользователя или отдельной вкладке: блок «Смены в ETL» (таблица из `getRecentCompletedJobs`) и при необходимости «Клиенты ETL» для справки.

**Результат:** админ видит данные ETL рядом с нашими; логика начисления и штрафов пока не меняется.

## Фаза 2: Справочник клиентов из ETL

**Цель:** использовать `mg_clients` при отображении истории и в настройках квестов (название клиента вместо только `client_id`).

**Варианты:**

- **A) Чтение по требованию**  
  При запросе истории транзакций или квестов — для каждого `client_id` при необходимости один запрос к ETL (или batch) и подстановка имени клиента в ответ. Плюс: всегда актуальные данные. Минус: зависимость от доступности ETL при каждом запросе.

- **B) Кэш в нашей БД**  
  Таблица `etl_clients_cache` (id, external_id, name, updated_at). Периодическая джоба (cron) или ручной «Обновить из ETL» в админке заполняет/обновляет кэш. API дашборда и админки читают только нашу БД. Плюс: быстрые ответы и работа при недоступности ETL. Минус: задержка обновления.

**Задачи:**

1. Выбрать вариант (A или B) и при B — добавить миграцию для кэш-таблицы.
2. Реализовать получение списка/одного клиента из ETL (`EtlDataService`).
3. При варианте B: джоба или эндпоинт админки «Синхронизировать клиентов из ETL».
4. В API истории транзакций и в квестах при наличии `client_id` подставлять имя из ETL или из кэша.

**Результат:** в UI отображаются названия клиентов из ETL (напрямую или из кэша).

## Фаза 3: Начисление за смены из ETL (автоматизация)

**Цель:** обрабатывать завершённые смены из ETL как источник фактов и вызывать существующую логику начисления (и квестов), без дублирования.

**Задачи:**

1. **Джоба «синхронизация смен из ETL»**  
   - Запуск по расписанию (cron, например раз в 5–15 минут) или по триггеру.
   - Читает из `mygig_job_history` записи, удовлетворяющие критерию «завершённая сменa» и, при необходимости, «ещё не обработанные» (по дате обновления или флагу в ETL, если есть).
   - Для каждой записи: найти пользователя по `external_id` в нашей БД; если такой смены ещё нет (`transactions` с таким `source_ref` или отдельная таблица `processed_etl_shift_ids`), вызвать `recordShiftCompleted` с полями из ETL (монеты/часы, client_id, category, location и т.д.).

2. **Идемпотентность**  
   Ключ обработки — ID смены в ETL (или уникальная пара user_id + shift_id). Либо проверка наличия транзакции с `source_ref = etl_shift_id`, либо таблица `processed_etl_shift_ids(etl_shift_id PRIMARY KEY)` перед вызовом `recordShiftCompleted`.

3. **Ошибки и логи**  
   При отсутствии пользователя по `external_id` — лог/алерт, запись не обрабатывается (или в очередь повтора). Логировать количество обработанных и пропущенных записей за прогон.

4. **Конфиг**  
   Флаг/настройка «включить синхронизацию смен из ETL» (env или `system_settings`), чтобы можно было отключить без выката.

**Результат:** новые завершённые смены в ETL автоматически приводят к начислению монет и пересчёту квестов в нашей БД; ручной ввод в админке остаётся как запасной вариант.

---

## Фаза 4: Штрафы из ETL (если есть источник)

**Цель:** если в ETL есть данные об отменах (поздних) или no-show, регистрировать штрафы в нашей БД теми же правилами, что и при ручном вводе.

**Задачи:**

1. Уточнить в ETL наличие таблиц/полей: отмены бронирований, no-show, привязка к пользователю и смене.
2. Джоба или расширение джобы из фазы 3: выборка «новых» отмен/no-show из ETL, маппинг на `user_id` по `external_id`, проверка идемпотентности (например по `shift_external_id` в `strikes`), вызов логики создания штрафа (`POST /v1/rewards/strikes` или внутренний сервис).
3. Аналогично фазе 3: логи, обработка отсутствующего пользователя, конфиг «включить штрафы из ETL».

**Результат:** штрафы из ETL автоматически попадают в нашу БД и учитываются в уровнях.

## Фаза 5: Документы и доработки

- **mygig_documents** — при необходимости: проверка статуса документа при первом входе, отображение в профиле или админке. Требования уточняются по продукту.
- **Отключение моков** — когда ETL (или API основной системы) станет основным источником смен/штрафов: убрать или скрыть ручной ввод смен в админке для прода, оставив только для тестовых окружений при необходимости.
- **Мониторинг** — метрики/алерты: доля успешно обработанных смен из ETL, задержка между датой смены в ETL и появлением транзакции у нас.

## Технические заметки

- **Чтение ETL:** использовать существующий `EtlExplorerService` (ClickHouse HTTP) или выделить общий клиент в `EtlDataService`, чтобы не дублировать логику подключения и формата запросов.
- **Джобы:** реализовать в NestJS как scheduled tasks (`@nestjs/schedule`) или отдельный worker; конфиг расписания через env.
- **Безопасность:** доступ к ETL только с бэкенда; учётные данные только в env (как сейчас). Не пробрасывать сырые запросы к ETL с фронта.
- **Резерв:** при падении ETL или отключении интеграции приложение продолжает работать за счёт нашей БД и, при необходимости, ручного ввода в админке.

После утверждения маппинга (фаза 0) можно переходить к реализации фазы 1 в коде.
