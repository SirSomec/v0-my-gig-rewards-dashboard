# План: получение данных о сменах из сервиса TOJ

Доработка обеспечивает **загрузку смен из TOJ** (реального или мокового), сопоставление с пользователями по `external_id` и **начисление монет** через существующую логику без дублирования.

---

## Реализовано (синхронизация смен из TOJ)

- **Идемпотентность начисления:** в `recordShiftCompleted` (RewardsService) добавлен опциональный параметр `sourceRef`. При передаче перед созданием транзакции проверяется наличие записи `type = 'shift'` с таким `source_ref`; при наличии возвращается существующий `transactionId`. В новую транзакцию записывается `sourceRef`.
- **Клиент TOJ:** модуль `modules/toj/` — `TojClientService` вызывает `POST /job.find-many.query` с Basic Auth (env: `TOJ_BASE_URL`, `TOJ_USER`, `TOJ_PASSWORD`), фильтры `workerIds`, `statuses`, `updatedAtGte`, пагинация `limit`/`skip`.
- **Синхронизация:** `TojSyncService` — watermark в `system_settings` (ключ `toj_sync_last_updated_at`). При запуске: выборка пользователей с `external_id`, батчи по `TOJ_SYNC_WORKER_BATCH_SIZE` (200), запросы к TOJ с пагинацией; лимит смен за прогон `TOJ_SYNC_MAX_JOBS_PER_RUN` (1000). Для каждой смены: поиск user по `workerId`; пропуск, если дата смены < `user.created_at`; проверка `source_ref`; вызов `recordShiftCompleted(..., sourceRef: job._id)`. Первый запуск: watermark = сейчас − `TOJ_SYNC_INITIAL_DAYS_AGO` (7 дней).
- **Админ API:** `GET /v1/admin/toj-sync/status` — настроен ли TOJ, включена ли синхронизация; `POST /v1/admin/toj-sync/run` — запуск синхронизации, ответ `{ processed, skipped, errors, watermark? }`.
- **Админка:** на странице «Мок TOJ (смены)» блок «Синхронизация смен из TOJ» — статус, кнопка «Синхронизировать смены», отображение результата.
- **Переменные окружения:** см. `.env.example` и docker-compose (TOJ_BASE_URL, TOJ_USER, TOJ_PASSWORD, TOJ_SYNC_ENABLED, TOJ_SYNC_MAX_JOBS_PER_RUN, TOJ_SYNC_INITIAL_DAYS_AGO, TOJ_SYNC_WORKER_BATCH_SIZE, TOJ_SYNC_PAGE_SIZE).
- **Cron** не добавлен (при необходимости — `@nestjs/schedule` и вызов `TojSyncService.runSync()` по расписанию).
- **Поздняя отмена (штраф):** если смена перешла в статус `cancelled` менее чем за 24 ч до начала и **инициатор отмены — работник** (`meta.initiatorType = "worker"` в TOJ `job.update.command`), начисляется штраф «поздняя отмена» (запись в `strikes`, тип `late_cancel`), он отображается в активностях пользователя. Эндпоинт `POST /v1/admin/toj/process-late-cancel` (body: `jobId`, `workerId`, `jobStart`, `cancelledAt`, `initiatorType?`, `initiator?`) — вызов из вебхука TOJ или при получении данных об отмене. Для проверки используется `initiatorType === "worker"` (или `initiator === "worker"` для совместимости). При начислении пишется запись в аудит (`late_cancel_applied`). Идемпотентность: повторный вызов для той же смены не создаёт второй штраф.

## 1. Общая логика

1. **Источник данных** — TOJ API (или мок): `POST /job.find-many.query` с фильтрами (статусы «к начислению», при необходимости диапазон дат).
2. **Для каждой смены в ответе:**
   - Найти пользователя в нашей БД по `workerId` смены = `users.external_id`.
   - Если пользователя нет — логировать и пропускать смену (не падать).
   - Проверить **идемпотентность**: смена уже обработана, если есть транзакция `type = 'shift'` с `source_ref = job._id`. Если есть — пропустить.
   - Если смена новая — вызвать **начисление за смену** с полями из TOJ (часы, клиент, название и т.д.) и сохранить в транзакции `source_ref = job._id`.
3. **Запуск** — по расписанию (cron, например раз в 5–15 минут) и/или вручную из админки (кнопка «Синхронизировать смены из TOJ»).
4. **Конфиг** — флаг/переменная «включить синхронизацию из TOJ», чтобы можно было отключить без выката. URL TOJ и учётные данные — из env (тот же мок или боевой TOJ).

Итог: новые завершённые смены в TOJ один раз превращаются в транзакции и квесты у нас; повторный прогон ту же смену не дублирует.

---

## 2. Критерий «смена к начислению»

- В TOJ считаем подходящими смены только со статусом **`confirmed`** (по [TOJ-API.md](./TOJ-API.md)).
- Опционально: фильтр по дате обновления/создания (например, только смены за последние N дней), чтобы не опрашивать весь архив при первом запуске.

---

## 3. Маппинг TOJ → наша модель

| Поле TOJ (JobEntityOptional) | Назначение у нас |
|-----------------------------|------------------|
| `_id` | Идемпотентность: `source_ref` в транзакции. |
| `workerId` | Связь с пользователем: `users.external_id = workerId`. |
| `customName`, `spec` | `title` транзакции (например, `customName || spec`). |
| `clientId` | `client_id` транзакции (для квестов по клиенту). |
| `spec` (или отдельное поле категории) | `category` транзакции (для квестов по категории). |
| `hours` | Передаём в начисление — бонус считается: ceil(hours) × множитель × множитель уровня. |
| `start` / `finish` | Для логов и при необходимости фильтра по периоду. |

**Учёт только смен с даты регистрации в программе лояльности:** данные по пользователю засчитываем только начиная с момента создания учётной записи в нашем сервисе. При обработке смены после нахождения пользователя по `workerId` проверять: дата смены (`job.start` или при отсутствии — `job.createdAt`) должна быть **≥ `users.created_at`**. Если смена раньше даты регистрации — пропускать (не начислять). Поле `users.created_at` уже есть в схеме (timestamps).

Если в TOJ нет часов — можно передать только `coins` (например, 0 или расчёт по другой логике) и не передавать `hours`; тогда используется ручной ввод суммы (или нужно договориться правило по умолчанию).

## 4. Идемпотентность

- **Ключ обработки:** ID смены в TOJ = `job._id` (строка).
- **Проверка:** перед вызовом начисления проверить наличие транзакции с `type = 'shift'` и `source_ref = job._id` (по нашей БД). Если есть — смену не обрабатывать.
- **Запись:** при создании транзакции за смену обязательно сохранять `source_ref = job._id`.

Для этого текущий метод начисления нужно доработать: принимать опциональный `sourceRef` и:
- при вставке транзакции записывать его в `transactions.source_ref`;
- перед вставкой (если `sourceRef` передан) проверять: уже есть транзакция `type = 'shift'` с таким `source_ref` → тогда не создавать новую, вернуть существующую (или просто return без ошибки).

## 5. Компоненты и шаги реализации

### 5.1 Доработка начисления (RewardsService)

- В **`recordShiftCompleted`** добавить опциональный параметр **`sourceRef?: string`**.
- Если `sourceRef` передан:
  - Проверить: есть ли уже транзакция с `type = 'shift'` и `source_ref = sourceRef` (по всей БД или по пользователю — достаточно по БД, т.к. смена одна).
  - Если есть — вернуть существующую `transactionId` (идемпотентность), не создавать новую запись.
- При создании новой транзакции за смену — записывать в неё `sourceRef`, если передан.

### 5.2 Клиент TOJ (новый сервис/модуль)

- **TojClientService** (или слой в существующем модуле): метод типа **`findJobs(filters, options)`**, который:
  - дергает TOJ по `POST /job.find-many.query` с телом в формате TOJ (см. [TOJ-API.md](./TOJ-API.md));
  - использует Basic Auth из env (`TOJ_BASE_URL`, `TOJ_USER`, `TOJ_PASSWORD` или единый URL с креденшлами);
  - возвращает массив смен и при необходимости общее количество.
- Конфиг: `TOJ_BASE_URL` (в dev — мок, например `http://mock-toj:3010`), `TOJ_USER`, `TOJ_PASSWORD`. Если URL не задан — синхронизацию не запускать.

### 5.3 Синхронизация смен (джоба + вызов начисления)

- **TojSyncService** (или метод в AdminService/RewardsService):
  - Вызвать TOJ: выборка смен со статусом `confirmed` (и при необходимости фильтр по дате).
  - Для каждой смены:
    - Найти пользователя: `SELECT id, created_at FROM users WHERE external_id = job.workerId`.
    - Если не найден — лог и `continue`.
    - **Смена до регистрации:** если дата смены (`job.start` или `job.createdAt`) < `user.created_at` — пропустить (не засчитывать смены до подключения к программе лояльности).
    - Проверить идемпотентность: есть ли транзакция `type = 'shift'` и `source_ref = job._id`. Если да — `continue`.
    - Вызвать `recordShiftCompleted(userId, coins?, title, location?, clientId?, category?, hours?, sourceRef: job._id)` с полями из смены.
  - Вернуть/залогировать: сколько обработано, сколько пропущено (нет пользователя / уже есть), ошибки.

### 5.4 Запуск синхронизации

- **Вариант A — по расписанию:** NestJS `@nestjs/schedule`: cron, например каждые 10 минут, вызывает TojSyncService (только если включено по env).
- **Вариант B — вручную из админки:** эндпоинт `POST /v1/admin/toj-sync/run` (или `POST /v1/admin/sync-jobs`), который вызывает ту же логику синхронизации. Кнопка в админке «Синхронизировать смены из TOJ».
- Рекомендуется оба: cron для фона, ручной запуск для проверки и отладки.

### 5.5 Конфиг и безопасность

- Env: `TOJ_SYNC_ENABLED` (true/false), `TOJ_BASE_URL`, `TOJ_USER`, `TOJ_PASSWORD`. При отсутствии URL или отключённом флаге джоба не выполняется и ручной запуск возвращает ошибку «не настроено».
- Учётные данные TOJ только на бэкенде, не отдавать на фронт.

### 5.6 Поздняя отмена смены (штраф за отмену менее чем за 24 ч до начала)

**Правило:** если смена перешла в статус `cancelled` **менее чем за 24 часа до её начала** и при этом **инициатор отмены — работник** (`meta.initiatorType = "worker"` в TOJ при вызове `job.update.command`), в системе наград начисляется штраф «поздняя отмена» (`late_cancel`).

- **Где отображается:** штраф попадает в таблицу `strikes` и выводится в **активностях** пользователя (история начислений и штрафов в личном кабинете) как «Штраф: Поздняя отмена (смена #…)».
- **Источник данных:** в TOJ при обновлении смены (в т.ч. отмена) в теле `POST /job.update.command` передаётся объект `meta`: `initiatorType` (например `"worker"`, `"operator"`) и `initiator` (идентификатор). Для штрафа используется **initiatorType === "worker"**. Интеграция при получении события отмены (вебхук или опрос API) вызывает наш эндпоинт, передавая `initiatorType` из `meta`.
- **Эндпоинт:** `POST /v1/admin/toj/process-late-cancel` (X-Admin-Key). Тело: `{ jobId, workerId, jobStart, cancelledAt, initiatorType?, initiator? }` (даты в ISO 8601). Условие «инициатор — работник» проверяется по `initiatorType === "worker"` (или `initiator === "worker"` для совместимости). При выполнении условий штраф создаётся один раз, в аудит пишется действие `late_cancel_applied`.

## 6. Нагрузка и инкрементальность (сотни тысяч смен, тысячи на пользователя)

Чтобы не тянуть и не обрабатывать сотни тысяч смен за раз, нужны **инкрементальная выборка**, **пагинация** и **лимиты на прогон**.

### 6.1 Водяной знак (watermark) — только «новое»

- Хранить **дату последней успешной синхронизации** по `updatedAt` (например, в БД: таблица `sync_state` с ключом `toj_jobs`, полем `last_updated_at` или в env после каждого прогона).
- Каждый прогон запрашивать в TOJ только смены с **`updatedAt` ≥ last_updated_at** (формат TOJ: `filters.updatedAt = ["gte:ISO8601"]`). Так за один раз приходят только изменившиеся смены, а не весь архив.
- После успешной обработки порции обновлять watermark максимальным `updatedAt` среди обработанных (или временем окончания прогона).
- **Первый запуск:** задать начальный watermark (например, «сейчас минус 7 дней») или обрабатывать только последние N дней (см. ниже).

### 6.2 Пагинация запросов к TOJ

- В TOJ в запросе передавать **`options.limit`** (например, 200–500) и **`options.skip`**.
- За один прогон в цикле: запрос с `limit=500, skip=0` → обработать страницу → затем `skip=500`, и так далее.
- Прекращать цикл, когда: вернулось меньше `limit` смен (страницы кончились) или достигнут **лимит смен на прогон** (см. ниже). Так в память и в обработку не попадают сотни тысяч записей за раз.

### 6.3 Лимит смен за один прогон

- Ввести конфиг **`TOJ_SYNC_MAX_JOBS_PER_RUN`** (например, 500 или 1000).
- За один запуск (cron или ручной) обрабатывать **не более** этого числа смен. Остальные подтянутся следующими прогонами (watermark сохранит прогресс).
- Это ограничивает время работы джобы и нагрузку на БД и TOJ.

### 6.4 Фильтр по нашим пользователям (workerIds)

- В TOJ сотни тысяч смен по всем работникам; у нас — только пользователи с заведённым `external_id`. Чтобы не забирать «чужие» смены пагинацией по миллионам:
  - **Вариант A:** запрашивать с **`filters.workerIds = [список external_id наших пользователей]`**. Список наших `external_id` брать из БД (один раз за прогон). Если пользователей тысячи — разбить на **батчи** (например, по 100–200 `workerId` за запрос), для каждого батча делать свой запрос с пагинацией (limit/skip) и общим фильтром по `updatedAt`. Так нагрузка на TOJ и объём ответа предсказуемы.
  - **Вариант B:** не передавать `workerIds`, а фильтровать по `updatedAt` (и статусам) и на нашей стороне отбрасывать смены, у которых `workerId` нет в `users.external_id`. Минус: TOJ вернёт много лишних смен (всех работников), если за период много обновлений; плюс — один простой запрос. При сотнях тысяч смен вариант B даёт большую нагрузку на сеть и память, поэтому **предпочтительно A с батчами по workerIds**.

### 6.5 Рекомендуемые параметры (пример)

| Параметр | Значение | Назначение |
|----------|----------|------------|
| Watermark | хранить `last_updated_at` | Запрашивать только смены с `updatedAt >= last_updated_at`. |
| limit на страницу TOJ | 200–500 | Размер одной страницы в запросе к TOJ. |
| max jobs per run | 500–1000 | Максимум смен обработать за один запуск. |
| batch workerIds | 100–200 | Сколько `external_id` подставлять в один запрос (если запросов несколько — по батчам). |
| cron | каждые 10–15 мин | Частота фоновой синхронизации. |

### 6.6 Первый прогон (бэкфилл)

- Для самого первого запуска, когда watermark ещё нет, можно:
  - задать начальный `last_updated_at` = «сейчас − 7 дней» (или −30), чтобы не трогать весь архив; или
  - один раз запустить «полную» синхронизацию вручную с большим окном (например, 90 дней) и лимитом на прогон (например, 2000): обрабатывать по 500 смен за запрос, обновлять watermark, пока не обработаем 2000 или не кончатся данные за окно.
- Дальше обычная инкрементальная синхронизация по watermark подтянет только новые изменения.

### 6.7 Итог по нагрузке

- За один прогон: несколько запросов к TOJ (батчи по `workerIds` × страницы по `limit`), но не более `TOJ_SYNC_MAX_JOBS_PER_RUN` смен к обработке.
- В каждом запросе к TOJ: ограниченный объём (limit 200–500) и фильтр по `updatedAt` и при необходимости по `workerIds`.
- Память и CPU: обрабатываются только смены одной порции; watermark не даёт повторно тянуть одни и те же данные.

## 8. Изменение статуса смены в TOJ

Смена в TOJ может менять статус (например `pending` → `completed` → `confirmed` или `confirmed` → `cancelled`). В текущей реализации это учтено **частично**.

- **Учтено:**
  - Запрашиваются только смены со статусом **`confirmed`**. Смена, которая ещё не в статусе confirmed, не попадёт в выборку; когда в TOJ её переведут в confirmed и обновят `updatedAt`, она попадёт в следующий прогон по watermark и будет обработана один раз.
  - Идемпотентность по `source_ref`: одна и та же смена (один `job._id`) не начисляется дважды, даже если приходит в нескольких прогонах.

- **Не учтено (ограничения):**
  1. **Отмена/отклонение смены в TOJ** (например `confirmed` → `cancelled`): мы запрашиваем только `statuses: ['confirmed']`, поэтому после смены статуса такая смена перестаёт приходить. Начисление у нас **остаётся** — откат или отмена начисления при изменении статуса в TOJ **не реализованы**.
  2. **Изменение данных смены** (часы, название, дата) после того, как мы уже начислили: при повторном приходе смены (тот же `job._id`) мы только проверяем `source_ref` и пропускаем. **Обновление** уже созданной транзакции (пересчёт монет, правка полей) **не выполняется**.

При необходимости в будущем можно:
- добавить отдельный поток «отмена начисления»: запрашивать смены со статусами отмены и по `source_ref` отменять/реверсировать транзакцию;
- или запрашивать по `updatedAt` без фильтра по статусу и для уже известных `source_ref` сверять статус и при отмене в TOJ откатывать начисление (требует доработки модели транзакций и API отмены).

## 9. Ошибки и граничные случаи

- **TOJ недоступен** — залогировать, не падать (при следующем прогоне повторить).
- **Пользователь не найден по workerId** — логировать и пропускать смену (не создавать пользователя автоматически).
- **Смена до даты регистрации пользователя** — дата смены (`start` или `createdAt`) раньше `users.created_at`: не начислять, пропустить (засчитываем только смены после подключения к программе лояльности).
- **Дубликат по source_ref** — уже обработано, пропустить без ошибки.
- **Ошибка при recordShiftCompleted** (например, падение БД) — залогировать, при необходимости оставить смену на следующий прогон (идемпотентность по source_ref защитит от двойного начисления при повторе).

---

## 10. Связь с ETL

Тот же поток (смены → пользователь по external_id → начисление с source_ref) описан в [ETL-INTEGRATION.md](./ETL-INTEGRATION.md) для ETL. Источником фактов может быть **либо TOJ API, либо ETL** (или оба с разными source_ref-префиксами, чтобы не пересекаться). Имеет смысл вынести общую часть «проверка source_ref + вызов recordShiftCompleted» в один слой, а источник данных (TOJ или ETL) подставлять разным адаптером.
