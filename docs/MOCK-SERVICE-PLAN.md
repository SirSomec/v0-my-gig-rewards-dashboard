# План: моковый сервис для разработки

Моковый сервис нужен, чтобы во время разработки не зависеть от реального TOJ (и при необходимости от бэкенда Rewards) и иметь предсказуемые тестовые данные.

---

## Реализовано (мок TOJ)

- **Отдельный сервис** `mock-toj-service/` (Express), порт 3010. Запуск: `docker-compose up mock-toj` или из папки `mock-toj-service` — `npm start`.
- **Эндпоинты TOJ:** `POST /job.find-many.query`, `POST /job.count.query`, `POST /job.find-by-id.query` — Basic Auth из env (`MOCK_TOJ_USER`, `MOCK_TOJ_PASSWORD`).
- **Генерация из админки:** `POST /admin/generate-jobs` (защита по `X-Admin-Key` = `MOCK_TOJ_ADMIN_KEY`). Бэкенд проксирует вызов: `POST /v1/admin/mock-toj/generate` с телом `{ userId, count?, dateFrom?, dateTo? }` — подставляется `external_id` выбранного пользователя.
- **Админка:** раздел «Мок TOJ (смены)» — выбор пользователя (из списка с external_id), количество смен, период, кнопка «Сгенерировать смены»; блок «Сгенерированные смены» — таблица с просмотром списка (эндпоинт `GET /admin/jobs` в моке, проксируется через `GET /v1/admin/mock-toj/jobs`); блок «Синхронизация смен из TOJ» — запуск синхронизации из админки (см. [TOJ-SYNC-PLAN.md](./TOJ-SYNC-PLAN.md)).
- Переменные окружения: см. `.env.example` (MOCK_TOJ_*, в api — MOCK_TOJ_URL, MOCK_TOJ_ADMIN_KEY).

---

## 1. Цели

1. **Мок TOJ API (Jobs)** — отдавать фейковые смены в формате `job.find-many.query` / `job.count.query` / `job.find-by-id.query`, чтобы:
   - тестировать будущий поток «синхронизация смен из TOJ» (джоба или ручной вызов);
   - разрабатывать маппинг TOJ → наша модель (user by external_id, transaction, source_ref);
   - проверять пагинацию и фильтры без доступа к spellbook-qa.

2. **Опционально: мок Rewards API** — отдавать фейковые `me`, `transactions`, `quests`, `store`, `levels`, `strikes`, чтобы:
   - разрабатывать фронт дашборда без запущенного Nest + PostgreSQL (достаточно Next.js + мок).

В первом итерации разумно ограничиться **только моком TOJ**; мок Rewards при необходимости добавить отдельно.

---

## 2. Что мокаем (первая итерация — TOJ)

| Эндпоинт TOJ | Назначение |
|--------------|------------|
| `POST /job.find-many.query` | Список смен с фильтрами и пагинацией (limit/skip/sort). |
| `POST /job.count.query` | Количество смен по тем же фильтрам. |
| `POST /job.find-by-id.query` | Одна смена по `id`. |

Формат запроса/ответа — по [TOJ-API.md](./TOJ-API.md): обёртка `{ "data": ..., "error": null }`, тело запроса — `{ "data": { "filters", "projection", "options" } }`.

Минимальная поддержка фильтров в моке (для реалистичности):
- `workerIds` — оставить только смены с `workerId` из списка;
- `statuses` — по полю `status`;
- `start` / `finish` — диапазон дат (если передан массив из двух элементов в формате gte/lte);
- `ids` — по списку ID смен.

Остальные фильтры можно игнорировать (всегда возвращать полный мок-набор, затем применять только эти). Пагинация: `options.limit`, `options.skip`, `options.sort` — применять к отфильтрованному списку.

## 3. Где размещать мок

**Вариант A — модуль внутри NestJS (рекомендуется для TOJ)**  
- В том же приложении (`nestjs-service`) добавляется модуль, например `MockTojModule`.
- Маршруты: `POST /mock/toj/job.find-many.query`, `POST /mock/toj/job.count.query`, `POST /mock/toj/job.find-by-id.query` (или единый префикс `/:path`, если TOJ всегда вызывает по path).
- Включение по env: `MOCK_TOJ_ENABLED=true`. Когда позже появится TOJ-клиент, он будет читать `TOJ_BASE_URL`; в dev можно задать `TOJ_BASE_URL=http://localhost:3001/mock/toj`.
- Плюсы: один процесс, один порт, не нужно поднимать второй сервис. Минус: мок живёт в основном приложении (можно вынести в отдельный пакет при желании).

**Вариант B — отдельный процесс (скрипт или мини-сервер)**  
- Отдельная папка, например `mock-service/`, с простым Express (или Next.js API routes на другом порту).
- Реализуются те же три эндпоинта в формате TOJ. В dev запускаем два процесса: Nest и mock (например, порт 3010). В `.env` задаём `TOJ_BASE_URL=http://localhost:3010`.
- Плюсы: полное разделение продакшн-кода и мока. Минусы: два процесса в dev, нужно описать запуск в README/docker-compose.

**Рекомендация:** начать с **варианта A** (NestJS-модуль), чтобы не усложнять окружение. При необходимости позже вынести мок в отдельный репозиторий/сервис.

## 4. Источник мок-данных

- **Статический набор смен** в коде (массив объектов в формате `JobEntityOptional` из TOJ-API.md).
- Поля обязательно заполнить для сценариев начисления: `_id`, `status`, `workerId`, `employerId`, `clientId`, `workplaceId`, `coordinatorId`, `spec`, `customName`, `start`, `finish`, `startFact`, `finishFact`, `hours`, `salaryPerHour`, `paymentPerJob`, `createdAt`, `updatedAt`. Часть полей можно задать `null`/пустыми.
- Количество записей: порядка 10–20 с разными датами, 2–3 разными `workerId` (соответствующими тестовым `external_id` пользователей в нашей БД), статусы `completed`/`confirmed`.
- Опционально: сценарии через query или env, например `?scenario=empty` или `MOCK_TOJ_SCENARIO=empty`, чтобы возвращать пустой список (проверка «нет смен»).

1. **Подтвердить объём:** только TOJ (job.find-many, count, find-by-id) в первой итерации или сразу плюс мок Rewards API для фронта?
2. **Размещение:** мок внутри NestJS (вариант A) или отдельный процесс (вариант B)?
3. **Тестовый пользователь:** какой `external_id` у тестового пользователя в вашей БД (или создать одного)? Его нужно подставить в мок-смены в поле `workerId`, чтобы при маппинге «смена → user по external_id» находился наш user.
4. **Авторизация:** в реальном TOJ — Basic Auth. В моке в dev достаточно принимать любой Basic или не проверять авторизацию; нужно ли для мока имитировать проверку (например, фиксированная пара login/password в env)?

## 6. Ограничения мока

- **Нет персистентности:** любые вызовы create/update/cancel в TOJ в моке не реализуем (или возвращают 200 без изменения состояния). Мок только отдаёт заранее заготовленный список.
- **Фильтрация:** имеет смысл реализовать только часть фильтров (workerIds, statuses, ids, start/finish, limit/skip/sort), чтобы сценарии «смены по worker» и «пагинация» работали. Остальные фильтры можно не учитывать (игнорировать в моке).
- **Нет бизнес-логики:** валидация полей, правила переходов статусов и т.п. не повторяются; мок просто возвращает JSON.
- **Один набор данных:** для смены сценария (например, «пусто») — через query/env (scenario), без отдельной «базы» в памяти.
- **Только для разработки/тестов:** мок не должен включаться в prod (env `MOCK_TOJ_ENABLED=false` или отсутствует). В коде явно помечать модуль/маршруты как dev-only (комментарии, проверка env при регистрации).

## 7. Пошаговый план реализации (только TOJ, вариант A)

| Шаг | Задача |
|-----|--------|
| 1 | Добавить в NestJS модуль `MockTojModule` с контроллером, маршруты под префиксом `/mock/toj` (например, `POST /job.find-many.query` и т.д.). Регистрировать модуль только если `MOCK_TOJ_ENABLED === 'true'`. |
| 2 | Описать тип ответа TOJ (JobEntitiesOptionalWrapper, JobEntityOptional) в TypeScript (или взять из общей спецификации OpenAPI, если подключаем). |
| 3 | Завести статический массив мок-смен (5–20 записей), подставить в часть записей `workerId` = тестовый external_id. |
| 4 | В обработчике `job.find-many.query`: парсить `body.data.filters` и `body.data.options`, фильтровать по workerIds/statuses/ids/start/finish, применять sort/skip/limit, возвращать `{ data: [...], error: null }`. |
| 5 | В обработчике `job.count.query`: те же фильтры, вернуть `{ data: { total: number }, error: null }`. |
| 6 | В обработчике `job.find-by-id.query`: по `body.data.id` найти смену в том же массиве, вернуть `{ data: job \| null, error: null }`. |
| 7 | Обновить README / .env.example: описать `MOCK_TOJ_ENABLED`, `TOJ_BASE_URL` для dev (когда TOJ-клиент появится). |
| 8 | При необходимости добавить сценарий `scenario=empty` (query или env), при котором find-many и count возвращают пустой список. |

После этого при добавлении в проект TOJ-клиента (вызовы на `TOJ_BASE_URL`) в dev можно будет выставить `TOJ_BASE_URL=http://localhost:3001/mock/toj` и использовать мок вместо реального TOJ.

## 8. Дальнейшие возможности (вне первой итерации)

- Мок Rewards API (me, transactions, quests, store, levels, strikes) через Next.js API routes или отдельный мини-сервер для разработки фронта без бэкенда.
- Мок ETL-таблиц (mygig_job_history, mg_clients) — если решите тестировать фазу 1–2 ETL-интеграции без реального ClickHouse.
- Генерация мок-смен по правилам (faker, несколько дат, случайные workerId) при старте приложения для разнообразия данных.

После ответов на пункт 5 можно переходить к реализации по шагам из раздела 7.
