# MyGig Rewards Dashboard — функционал и механика

Краткое описание сервиса, основные экраны, механики начисления и обмена наград.

---

## 1. Назначение сервиса

**MyGig Rewards Dashboard** — мобильно-ориентированный веб-дашборд для гиг-работников (смены на складах, мероприятиях, доставке и т.п.). Сервис показывает:

- баланс внутренней валюты (**Gig Coins**);
- прогресс по уровню лояльности и условия перехода на следующий уровень;
- квесты (ежедневные и еженедельные цели) и награды за них;
- историю начислений монет;
- магазин обмена монет на скидки, бустеры, мерч и подарочные карты.

Цель — геймификация работы: больше смен и достижений → больше монет и уровень → больше преимуществ и наград.

---

## 2. Основные экраны (вкладки)

| Вкладка   | Назначение |
|-----------|------------|
| **Главная** | Прогресс уровня, квесты, последняя активность (начисления). |
| **История** | Полный список начислений монет (смены, бонусы, квесты). |
| **Магазин** | Каталог товаров/наград за монеты, кнопка обмена. |
| **Уровни**  | Все уровни лояльности, перки, пороги по сменам, текущий и открытые. |

Навигация — нижняя панель (Bottom Nav) с переключением вкладок; контент переключается с анимацией (framer-motion).

## 3. Функциональные блоки

### 3.1 Шапка (Header)

- **Аватар** и **имя пользователя**.
- **Бейдж уровня** (например, «Серебряный партнёр»).
- **Баланс Gig Coins** (отображается числом с иконкой монеты).
- Переключатель **светлая/тёмная тема**.
- Кнопка **уведомлений** (иконка с индикатором).

Данные пользователя, баланс и прогресс уровня приходят с API (`GET /v1/rewards/me`).

### 3.2 Прогресс уровня (Level Progress)

- **Текущий уровень** → **следующий уровень** (названия).
- **Прогресс-бар**: прогресс до **порога следующего уровня** — отображается «X / Y смен», где X — выполнено смен, Y — порог смен **следующего** уровня (сколько всего нужно для перехода). Шкала заполняется по мере приближения к Y.
- Подпись: «Ещё N до [название следующего уровня]» (N = Y − X). На максимальном уровне — «Максимальный уровень».
- **Рейтинг надёжности**: значение от 0 до 5 (дробное), по умолчанию 4. Отображается шкала «X.X / 5» и подпись о том, что рейтинг растёт за выполнение смен и снижается за прогулы и поздние отмены. Величины прироста и снижения задаются в админке (настройки рейтинга).
- Раскрывающийся блок **«Текущие преимущества»** с перками текущего уровня (бонусы %, множители монет, приоритет смен и т.д.).

Механика: переход на следующий уровень требует выполнения **порога смен следующего уровня**. При достижении порога пользователь переходит на следующий уровень, счётчик смен обнуляется. Уровень определяется **только количеством смен** (и ручным назначением в админке); **рейтинг надёжности** учитывается отдельно и в перспективе может влиять на механику учёта смен для уровня. Данные для прогресса приходят с API: `shiftsCompleted`, `nextLevelShiftsRequired` (порог следующего уровня), `shiftsRemaining`, `reliabilityRating`.

### 3.3 Квесты (цели по периодам и единоразовые)

- Блоки на главной: **Ежедневные цели**, **Еженедельные цели**, **Ежемесячные цели**, **Единоразовые цели**.
- У каждого квеста — название, описание, прогресс `progress/total` (смены или часы), награда в монетах, тип периода (`daily` / `weekly` / `monthly`). Квест может быть помечен как **единоразовый** (пользователь выполняет его только один раз).
- Визуальное выделение выполненных квестов (зелёная подсветка, зачёркивание). Для невыполненных — прогресс-бар и счётчик.

**Периоды:**

- **Ежедневные** — обновляются раз в сутки (UTC); невыполненные сгорают.
- **Еженедельные** — календарная неделя (пн–вс по UTC); в понедельник прогресс сбрасывается.
- **Ежемесячные** — календарный месяц; в начале месяца — новый период.
- **Единоразовые** — отдельный признак квеста (любой период); пользователь может выполнить такой квест только один раз.

**Типы условий выполнения** (задаются в админке, конфиг в `condition_config`):

| Тип | Описание | Параметры конфига |
|-----|----------|-------------------|
| `shifts_count` | Количество смен за период | `total` — целевое число смен |
| `bookings_count` | Количество **забронированных** смен за период (статус booked в TOJ; факт бронирования сохраняется и не обнуляется при смене статуса) | `total` — целевое число бронирований |
| `shifts_count_client` | Смены в конкретном клиенте (бренде) | `total`, `clientId` |
| `shifts_count_clients` | Смены в нескольких клиентах | `total`, `clientIds[]` |
| `shifts_count_category` | Смены в конкретной категории (профессии) | `total`, `category` |
| `hours_count` | Сумма отработанных часов за период | `totalHours` |
| `hours_count_client` | Часы в конкретном клиенте | `totalHours`, `clientId` |
| `hours_count_clients` | Часы в нескольких клиентах | `totalHours`, `clientIds[]` |
| `shifts_series` | Серия смен подряд без прогулов и поздних отмен: целевое число смен подряд, при пропуске смены или штрафе прогресс сбрасывается | `total` — целевое число смен подряд |
| `manual_confirmation` | Ручное подтверждение выполнения квеста администратором (кнопка «Подтвердить» в админке) | без параметров |

При записи смены (админка или API) можно передать **клиент** (`client_id`), **категорию** (`category`) и **часы** (`hours`) — они сохраняются в транзакции и учитываются при расчёте прогресса по квестам. Квест можно настроить на **автоотключение в конце периода** (поля `active_from` / `active_until`).

Механика: при `progress >= total` (или по часам — достижении `totalHours`) квест считается выполненным, начисляются монеты. Логика на бэкенде; фронт показывает период и счётчики.

### 3.4 История начислений (Earning History)

- Список записей: **название** (смена/бонус/квест/покупка), **локация**, **дата**, **+N** или **−N монет**.
- Типы записей: `shift` (смена), `bonus` (бонус), `quest` (награда за квест), `redemption` (покупка в магазине — отображается красным со знаком минус), **`strike`** (штраф: прогул или поздняя отмена — с указанием **за какую смену** получен штраф, по `shift_external_id`).

На главной показываются первые несколько записей; на вкладке «История» — полный список (транзакции и штрафы в одном потоке, отсортированы по дате). Кнопка «Все» ведёт на вкладку «История». API: `GET /v1/rewards/transactions`, `GET /v1/rewards/strikes`.

### 3.5 Магазин наград (Redemption Store)

- Сетка товаров с категорией, названием, описанием и **ценой в монетах**.
- Категории (по мокам): скидки, бустеры, мерч, подарки.
- **Лимит тиража (общее количество):** у товара может быть задан `stock_limit` — максимальное число выкупов по этому товару (все пользователи). При заданном лимите отображается «Осталось N из M»; при исчерпании (N = 0) — «Нет в наличии», кнопка «Купить» неактивна. Без лимита (пусто) — товар в неограниченном количестве.
- Кнопка обмена: активна, если баланс ≥ цены **и** товар в наличии (остаток по лимиту > 0 или лимит не задан).

Механика: списание монет и выдача награды выполняются бэкендом; при покупке проверяются баланс и остаток по тиражу (считаются заявки со статусами pending и fulfilled); при исчерпании тиража API возвращает ошибку «Товар закончился».

### 3.6 Уровни лояльности (Levels View)

- Карточки всех уровней: название, **порог смен для перехода на этот уровень** (подпись «для перехода: X смен»; у базового уровня — «базовый уровень»), список перков.
- Состояния: **текущий** (подсветка), **открыт** (галочка), **заблокирован** (иконка замка, приглушённый вид). Уровень считается открытым, если выполнено смен ≥ его порога.

**Таблица уровней:**

| Уровень | Способ получения | Рейтинг надёжности* | Бонусы |
|---------|------------------|---------------------|--------|
| **Бронзовый новичок** | Базовый уровень: выдаётся **изначально без условий** (порог смен = 0). | — | Базовая ставка; стандартное расписание; начисление монет 1x. |
| **Серебряный партнёр** | Переход при **10** завершённых смен (порог следующего уровня). | 0–5 | +5% бонус за каждую смену; 2x монеты за смены в выходные; приоритет выбора смен (ранний доступ). |
| **Золотой партнёр** | Переход при **25** завершённых смен. | 0–5 | +10% бонус за смены; мгновенные выплаты (без периода ожидания); 3x монеты в выходные; гибкий график (индивидуальное планирование). |
| **Платиновый элит** | Переход при **50** завершённых смен. | 0–5 | +15% бонус за смены; VIP-поддержка; 5x монеты в выходные; эксклюзивные задания; бесплатный мерч. |

\* У каждого пользователя хранится **рейтинг надёжности** (0–5, дробное, по умолчанию 4). Он растёт за выполнение смен и снижается за прогулы и поздние отмены; настройки задаются в админке. В перспективе рейтинг может влиять на учёт смен для уровня. Подробнее — п. 4.1.

## 4. Механики работы (сводка)

### 4.1 Рейтинг надёжности, прогулы и поздняя отмена

Чтобы учитывать **надёжность** работника, введён **рейтинг надёжности** (0–5, дробное значение). По умолчанию у каждого пользователя рейтинг **4**.

**Штрафные события** (записываются в таблицу `strikes` для истории и влияют на рейтинг):

| Событие | Условие | Влияние на рейтинг |
|--------|---------|---------------------|
| **Прогул (no-show)** | Забронирована смена, работник не явился и не отменил её | Снижение на величину из настройки «Снижение за прогул» (админка) |
| **Поздняя отмена** | Отмена забронированной смены менее чем за **24 часа** до начала смены | Снижение на величину из настройки «Снижение за позднюю отмену» (админка) |

**Правила рейтинга:**

- Рейтинг хранится в `users.reliability_rating`, отображается как дробное число (например 4.2).
- **За выполненную смену** рейтинг **увеличивается** на величину из настройки «Прирост рейтинга за смену» (админка → Настройки → Рейтинг надёжности); верхняя граница 5.
- **За прогул или позднюю отмену** рейтинг **уменьшается** на соответствующую величину из настроек (отдельные параметры для прогула и для поздней отмены); нижняя граница 0.
- **При ручном снятии штрафа** (админка): рейтинг **возвращается** на величину снижения (восстановление до 5 при необходимости).
- **При переходе смены в статус «подтверждена»** (например из TOJ): штраф по этой смене снимается, рейтинг восстанавливается на величину снижения; при начислении смены дополнительно применяется прирост рейтинга за выполненную смену.
- Отмена смены **за 24 часа и более** до старта — **без штрафа**.
- Пользователь видит в **истории** за какую смену получен штраф: записи вида «Штраф: прогул (смена #ID)» / «Штраф: поздняя отмена (смена #ID)» по полю `shift_external_id`.

**Уровень лояльности** определяется количеством **засчитанных в прогресс уровня смен** (и ручным назначением в админке). Понижение уровня из‑за лимитов штрафов за неделю/месяц **больше не применяется**. Рейтинг надёжности влияет на механику учёта смен и повышение уровня (см. ниже).

**Настройки в админке** (Настройки → Рейтинг надёжности):

- Прирост рейтинга за выполненную смену (например 0.1).
- Снижение рейтинга за прогул (no_show) (например 0.2).
- Снижение рейтинга за позднюю отмену (например 0.2).
- **Минимальный рейтинг для учёта смены в прогресс уровня** (`reliability_min_rating_to_count_shift_for_level`, 0–5; 0 = без ограничения). Если текущий рейтинг пользователя **ниже** этого порога на момент фиксации смены, смена **не увеличивает** `shifts_completed`, но монеты и прирост рейтинга за смену начисляются.
- **Минимальный рейтинг для автоматического повышения уровня** (`reliability_min_rating_to_upgrade_level`, 0–5; 0 = без ограничения). При выполнении порога смен следующего уровня повышение произойдёт только если текущий рейтинг пользователя **не ниже** этого порога.

**Реализация на бэкенде (фактическая логика):**

- В таблице `users` хранится `reliability_rating` (real, 0–5, по умолчанию 4). В `system_settings` используются ключи:
  - `reliability_rating_increase_per_shift`;
  - `reliability_rating_decrease_no_show`;
  - `reliability_rating_decrease_late_cancel`;
  - `reliability_min_rating_to_count_shift_for_level`;
  - `reliability_min_rating_to_upgrade_level`.
- При записи смены (`recordShiftCompleted`):
  - рейтинг увеличивается на величину из настроек (cap 5);
  - если текущий рейтинг **до прироста** ≥ `reliability_min_rating_to_count_shift_for_level`, `shifts_completed` увеличивается на 1; иначе `shifts_completed` не меняется.
- При регистрации штрафа (`registerStrike`) рейтинг уменьшается на величину по типу штрафа (floor 0).
- При снятии штрафа (ручное или по смене в confirmed) вызывается восстановление рейтинга (`restoreReliabilityRatingForStrikeRemoval`); при confirmed затем фиксируется смена (прирост рейтинга за смену и, при достаточном рейтинге, учёт смены в прогресс уровня).
- При пересчёте уровня (`recalcUserLevel`):
  - вычисляется максимальный уровень по `shifts_required`, где `shifts_required <= shifts_completed`; при отсутствии подходящего уровня выбирается базовый (минимальный `sort_order`);
  - ручное назначение уровня админом не понижается пересчётом (обновление только при повышении `sort_order`);
  - при автоматическом повышении уровня (новый уровень с большим `sort_order`) дополнительно проверяется, что текущий рейтинг ≥ `reliability_min_rating_to_upgrade_level`; если рейтинг ниже порога, повышение **не выполняется** до улучшения рейтинга;
  - при автоматическом повышении счётчик `shifts_completed` обнуляется, при первоначальном назначении базового уровня — сохраняется текущее значение.

### 4.2 Остальные механики

| Механика | Описание |
|----------|----------|
| **Начисление монет** | За смену, бонусы и выполнение квестов. Количество задаётся в данных (сейчас мок). |
| **Уровень лояльности** | Зависит от числа **засчитанных в прогресс уровня смен** (`shifts_completed`); **переход на следующий уровень требует выполнения порога смен следующего уровня** и, при заданном пороге, **достаточного рейтинга надёжности**. При достижении порога и выполнении условия по рейтингу — переход и **обнуление счётчика смен**. Ручное назначение уровня в админке не перезаписывается автоматическим пересчётом (пересчёт только повышает уровень). Базовый уровень (минимальный sort_order) имеет порог 0 и выдаётся изначально без условий. **Рейтинг надёжности** (0–5) хранится отдельно и влияет на учёт смен для уровня и допуск к повышению; понижение уровня из‑за лимитов штрафов за неделю/месяц отключено. |
| **Квесты** | Ежедневные, еженедельные, ежемесячные и единоразовые цели; прогресс по условиям (смены, часы, клиент, категория); при выполнении — начисление монет. |
| **Магазин** | Трата монет на товары; проверка баланса и остатка по лимиту тиража (stock_limit) на бэкенде; при исчерпании тиража покупка запрещена. В каталоге API возвращает stockLimit и redeemedCount; фронт показывает «Осталось N из M» и «Нет в наличии». |

## 5. Данные и интеграция

- Данные дашборда (пользователь, баланс, уровень, прогресс, квесты, магазин, история) загружаются с **нашего API** через `lib/rewards-api.ts` и `hooks/use-rewards-dashboard.ts`. Моки в `page.tsx` не используются.
- **Бэкенд** в `nestjs-service/`: реализованы схема БД (Drizzle), миграция со справочником уровней, API для дашборда: `GET /v1/rewards/me` (в ответе: баланс, уровень, **nextLevelShiftsRequired**, **reliabilityRating** 0–5), `GET /v1/rewards/transactions`, `GET /v1/rewards/quests`, `GET /v1/rewards/store` (в ответе по каждому товару: `stockLimit`, `redeemedCount` — для отображения остатка тиража), `POST /v1/rewards/redemptions`. Текущий пользователь задаётся через `?userId=` или `DEV_USER_ID` (до этапа авторизации).
- **TOJ sync (архитектурно):** в модуле `toj` SQL/Drizzle-запросы вынесены из `TojSyncService` в отдельный `TojSyncRepository`; сервис синхронизации работает через методы репозитория (watermark, last run, выборка пользователей с `external_id`, проверка идемпотентности транзакции смены). Функциональное поведение синка не изменено.
- **Rewards/Admin (архитектурно):** в модулях `rewards` и `admin` сервисы переведены на доступ к БД через репозитории (`RewardsRepository`, `AdminDbRepository`) через DI; прямой доступ к `drizzleProvider` из сервисов убран. Функциональное поведение API не изменено.
- **Глубокое вынесение SQL:** в `AdminAuthService` запросы к таблице админ-пользователей полностью выполняются через `AdminDbRepository`; в `RewardsService` через `RewardsRepository` вынесены чтение `system_settings`, операции `strikes` и обновление `reliability_rating`; в `AdminService` через `AdminDbRepository` вынесены операции с `system_settings`, запись `audit_log` и lookup `external_id` пользователя для TOJ mock.
- **Дополнительный этап рефакторинга репозиториев:** в `AdminService` через `AdminDbRepository` также вынесены операции создания/поиска пользователя, смены уровня, ручных начислений/списаний и снятия штрафа; бизнес-правила и аудит остаются в сервисе.
- **Продолжение рефакторинга Admin:** в `AdminDbRepository` вынесены SQL-блоки для списка/деталей пользователей, списка/обработки redemption, CRUD магазина, CRUD квестов и операций с группами/участниками; `AdminService` в этих сценариях выполняет в основном оркестрацию и валидацию.
- **Текущее состояние слоя доступа к БД:** `AdminService` и `AdminAuthService` переведены на репозиторный доступ полностью; в `RewardsService` дополнительно вынесены в `RewardsRepository` read-операции профиля и истории (`recordPageView`, `getMe`, `getStrikes`, `getTransactions`), остальные блоки `rewards` продолжают миграцию.
- **Фронт** подключается к API: данные загружаются с бэкенда (`lib/rewards-api.ts`, `hooks/use-rewards-dashboard.ts`), отображаются скелетоны при загрузке и сообщение об ошибке при недоступности API. В магазине отображается остаток по тиражу («Осталось N из M»), при исчерпании — «Нет в наличии» и неактивная кнопка; кнопка «Купить» вызывает `POST /v1/rewards/redemptions` и обновляет баланс после успешного обмена.
- **Авторизация (dev-режим):** при наличии `NEXT_PUBLIC_DEV_USER_ID` фронт вызывает `POST /v1/auth/dev-login` с телом `{ userId }`, получает JWT и сохраняет его в localStorage; все запросы к API отправляют заголовок `Authorization: Bearer <token>`. В шапке доступна кнопка «Выйти» (сброс токена). Бэкенд: эндпоинт `POST /v1/auth/dev-login`, опциональная JWT-стратегия; rewards-эндпоинты принимают userId из JWT или из query/config.
- **Авторизация через MyGig (опционально):** если задан `NEXT_PUBLIC_MYGIG_API_URL`, доступен вход по номеру телефона и коду из СМС (API MyGig: `POST /auth/phone`, `POST /auth/code`, `GET /user/profile`). При первом успешном входе фронт вызывает наш бэкенд `POST /v1/auth/ensure-user` с заголовком `X-Internal-Secret` (значение из `REWARDS_INTERNAL_SECRET` в корневом `.env`); бэкенд создаёт или обновляет пользователя в БД и возвращает JWT для доступа к дашборду. Секрет `REWARDS_INTERNAL_SECRET` должен быть одинаковым в окружении app и api (корневой `.env`). Переменная задаётся в корневом `.env`; при её отсутствии вызов ensure-user вернёт ошибку с подсказкой.
- Для локальной разработки задают `NEXT_PUBLIC_REWARDS_API_URL` и `NEXT_PUBLIC_DEV_USER_ID` (см. `.env.example`). Для входа через MyGig дополнительно: `NEXT_PUBLIC_MYGIG_API_URL`, `REWARDS_INTERNAL_SECRET`. На бэкенде — `DEV_USER_ID`, при использовании JWT — `JWT_SECRET` (и опционально `JWT_EXPIRE`).
- **Админ-API** (заголовок `X-Admin-Key`): управление магазином — `GET/POST/PATCH/DELETE /v1/admin/store-items`; справочник уровней — `GET /v1/admin/levels`, `PATCH /v1/admin/levels/:id` (для базового уровня порог смен должен быть 0); настройки рейтинга надёжности — `GET /v1/admin/settings/reliability-rating`, `PATCH /v1/admin/settings/reliability-rating`; квесты — `GET/POST/PATCH/DELETE /v1/admin/quests`; заявки — `GET /v1/admin/redemptions` (параметры: `status`, `search`, `dateFrom`, `dateTo`, `page`, `pageSize`; ответ: `{ items, total, page, pageSize }`), `PATCH /v1/admin/redemptions/:id`, `POST /v1/admin/redemptions/bulk-update` (массовое выполнение/отмена); пользователи — `GET /v1/admin/users`, `GET /v1/admin/users/:id`, `PATCH /v1/admin/users/:id` (body: `levelId` — ручное изменение уровня, при этом shifts_completed обнуляется); ручные начисления — `POST /v1/admin/transactions` (body: `userId`, `amount`, `type: manual_credit | manual_debit`, `title?`, `description?`); штрафы — `POST /v1/admin/strikes` (мок: добавить штраф), `POST /v1/admin/strikes/:id/remove` (body: `reason?` — снять штраф с причиной, восстановление рейтинга); аудит — `GET /v1/admin/audit-log` (параметры: `page`, `pageSize`, `action`, `entityType`); мок смен — `POST /v1/admin/shifts/complete` (тело: `userId`, `coins`, `title?`, `location?`, `clientId?`, `category?`, `hours?`).
- Для продакшена нужно:
  - API пользователя (профиль, уровень, баланс);
  - API истории начислений;
  - API квестов и обновления прогресса;
  - API магазина (каталог, списание монет, выдача наград);
  - при необходимости — WebSocket/поллинг для обновления баланса и уведомлений.

## 6. Функционал администратора

Админ-панель предназначена для управления контентом, настройкой правил и ручными операциями по пользователям. **Реализована как отдельное приложение** (`admin-app/`), запускается на отдельном порту (по умолчанию 3002). Доступ — по паролю (`ADMIN_PANEL_PASSWORD` в `.env`) или без пароля в режиме разработки; запросы к API отправляются с заголовком `X-Admin-Key` (значение из `ADMIN_SECRET` / `NEXT_PUBLIC_ADMIN_SECRET`). В админке есть ссылка на основной дашборд и возможность открыть кабинет от имени пользователя (`NEXT_PUBLIC_DASHBOARD_URL`).

### 6.1 Управление магазином наград

- **Каталог товаров**: список всех позиций с фильтром по категории и статусу (активен/скрыт/закончился). **Реализовано:** страница `/admin/store` со списком товаров, кнопки «Добавить товар», «Изменить», «Удалить» (с подтверждением). API: `GET/POST/PATCH/DELETE /v1/admin/store-items` (DELETE — мягкое удаление).
- **Создание и редактирование товара**:
  - название, описание, категория (скидка, бустер, мерч, подарочная карта и т.д.);
  - цена в Gig Coins;
  - лимит количества (если ограниченный тираж) или «без лимита»;
  - даты действия (с/по), возможность временно скрыть без удаления;
  - иконка/изображение (опционально).
  - **Реализовано:** диалог с полями название, описание, категория, цена, иконка, лимит количества, порядок, чекбокс «Товар активен».
- **Порядок отображения**: сортировка (drag-and-drop или приоритет) для вывода в приложении пользователя.
- **Видимость по группам** (опционально): показывать товар только определённым сегментам (например, по уровню лояльности, региону или тегу).

### 6.2 Управление квестами

**Глобальные квесты (для всех пользователей):**

- Список квестов с периодом (ежедневный / еженедельный / ежемесячный), признаком «единоразовый», окном активности и статусом (активен/выключен). **Реализовано:** страница `/admin/quests` — список, кнопки «Добавить квест», «Изменить», «Отключить».
- Создание и редактирование квеста:
  - название, описание;
  - **период**: ежедневный (`daily`), еженедельный (`weekly`), ежемесячный (`monthly`);
  - **единоразовый** — флаг (квест выполняется пользователем только один раз);
  - **отключить в конце периода** — при создании квест автоматически скрывается после окончания текущего дня/недели/месяца; опционально задаются даты «активен с» / «активен до»;
  - **тип условия** и конфиг:
    - количество смен за период (`shifts_count`) — цель: число смен;
    - количество забронированных смен за период (`bookings_count`) — цель: число бронирований (учитываются смены в статусе booked в TOJ; факт бронирования не обнуляется при смене статуса);
    - смены в конкретном клиенте (`shifts_count_client`) — цель, код клиента;
    - смены в нескольких клиентах (`shifts_count_clients`) — цель, список кодов клиентов (через запятую);
    - смены в конкретной категории/профессии (`shifts_count_category`) — цель, категория;
    - выполнение часов (`hours_count`) — цель в часах;
    - часы в конкретном клиенте (`hours_count_client`) — цель в часах, код клиента;
    - часы в нескольких клиентах (`hours_count_clients`) — цель в часах, список кодов клиентов;
    - **серия смен** (`shifts_series`) — целевое число смен подряд без прогулов и поздних отмен; при пропуске смены или новом штрафе прогресс сбрасывается;
    - **ручное подтверждение** (`manual_confirmation`) — выполнение квеста подтверждается администратором вручную (страница модерации квестов или карточка пользователя);
  - награда в монетах, иконка, привязка (все / группа).
- При записи смены (страница «Мок: смены и штрафы» или API) можно указать **клиент**, **категорию** и **часы** — они сохраняются в транзакции и учитываются при пересчёте прогресса квестов.

**Квесты для отдельных групп:**

- Группы определяются по правилам (уровень, регион, партнёр, тег). В квесте задаётся `target_type` (`all` | `group`) и при необходимости `target_group_id`. Пользователь видит глобальные квесты и квесты своей группы.

### 6.3 Ручные начисления и списания

- **Ручное начисление монет**:
  - выбор пользователя (поиск по ID, email, имени);
  - сумма монет;
  - тип: бонус, компенсация, промо, ручное начисление и т.п.;
  - комментарий (внутренний, для истории).
  - После подтверждения — зачисление на баланс и запись в историю с меткой «ручное».
- **Ручное списание**: уменьшение баланса с указанием причины (возврат, отмена награды, корректировка). Запись в историю.
- **Массовое начисление**: загрузка CSV (user_id или email, сумма, комментарий) или выбор по фильтру (уровень, регион) и начисление одной суммой/по правилу. Лимиты и подтверждение перед применением.

### 6.4 Пользователи и уровни

- **Поиск и карточка пользователя**: баланс, уровень, количество завершённых смен, **рейтинг надёжности (0–5)**, штрафы за 30 дней (список), последние начисления и обмены.
- **Ручное изменение уровня**: повышение или понижение (например, после разбора жалобы). При сохранении **счётчик смен пользователя обнуляется**. Причина и запись в аудит. **Реализовано:** в карточке пользователя выбор уровня из справочника и кнопка «Сохранить уровень»; запись в audit_log (action: user_level_change). API принимает `levelId` (number или строка).
- **Управление штрафными баллами**: просмотр списка штрафов (no-show, поздняя отмена), при необходимости — снятие штрафа с указанием причины (ошибка системы, форс-мажор). При снятии **рейтинг надёжности пользователя восстанавливается** на величину снижения (настройки в админке). Пересчёт уровня по сменам и пересчёт прогресса по квестам (в т.ч. для типа «серия смен»). **Реализовано (6.7):** в карточке пользователя список штрафов; у каждого активного штрафа кнопка «Снять»; диалог с полем «Причина снятия». API: `POST /v1/admin/strikes/:id/remove` (body: `reason?`). В БД у штрафа проставляются `removed_at` и `removal_reason`; вызывается восстановление рейтинга и пересчёт уровня/квестов.

### 6.5 Уровни лояльности (справочник)

- Список уровней с порогами смен (**для перехода на этот уровень**) и списком перков. Поля лимитов штрафов за неделю/месяц в справочнике уровней **не используются** (логика переведена на рейтинг надёжности).
- **Базовый уровень** (с минимальным sort_order) должен иметь **порог смен = 0** — он выдаётся изначально без условий. В админке при редактировании этого уровня нельзя сохранить порог > 0 (валидация на бэкенде и подсказка на фронте).
- Редактирование: название уровня, порог смен для перехода, порядок отображения, перки (JSON), множитель бонусов. Минимальный и максимальный уровни не удаляются.
- **Реализовано:** страница `/admin/levels` со списком уровней, кнопка «Изменить» открывает диалог с полями: название, порог смен (для базового уровня — подсказка и проверка 0), порядок отображения, множитель бонусов, перки (JSON). Поля лимитов штрафов за неделю/месяц в форме **скрыты** (рейтинг надёжности настраивается в разделе «Настройки»). API: `GET /v1/admin/levels`, `PATCH /v1/admin/levels/:id`.

### 6.6 Остальной функционал

- **История операций (аудит)**: фильтруемый лог ручных начислений/списаний, смен уровней, снятия штрафов (кто, когда, что изменил). **Реализовано (6.9):** таблица `audit_log` (admin_id, action, entity_type, entity_id, old_values, new_values, created_at). Записи создаются при: изменении уровня пользователя (`user_level_change`), ручном начислении/списании (`manual_transaction`), снятии штрафа (`strike_removed`). API: `GET /v1/admin/audit-log?page=&pageSize=&action=&entityType=`. Страница `/admin/audit` — таблица с пагинацией и фильтрами по действию и типу сущности.
- **Заявки на обмен**: список заявок на товары из магазина (ожидают обработки/выполнены/отменены); отметка о выдаче награды или отмене с причиной. **Реализовано:** страница `/admin/redemptions` — таблица с пагинацией (до 200 записей на страницу), фильтры по статусу, поиску (пользователь/товар) и периоду (дата от–до); массовый выбор ожидающих заявок и массовое выполнение/отмена с возвратом монет; диалог с заметкой и опцией «Вернуть монеты»; ссылка на карточку пользователя. API: `GET /v1/admin/redemptions`, `PATCH /v1/admin/redemptions/:id`, `POST /v1/admin/redemptions/bulk-update`.
- **Уведомления**: рассылка уведомлений пользователям или группам (текст, ссылка); шаблоны для типовых сообщений (новый квест, уровень, акция в магазине).
- **Аналитика для админа**: сводки по балансам, оборотам монет, популярности квестов и товаров, конверсия по уровням — для настройки наград и порогов.
